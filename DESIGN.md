# OS-Agent：面向小型操作系统源码的多证据智能分析系统设计

## 摘要

本项目面向“全国大学生计算机系统能力大赛-操作系统设计赛(全国)-OS功能挑战赛道”中的赛题“面向小型操作系统的分析比对智能体系统设计”。自 2021 年 OS Kernel 赛道开展以来，参赛队伍围绕 RISC-V 等架构积累了大量风格各异的小型操作系统代码仓库，包括 rCore/uCore 变体、独立微内核、面向 Linux ABI 的兼容内核以及多种教学框架演进版本。到 2026 年，历史代码已经形成规模化存量。如何在新作品提交后自动理解其技术结构、定位其来源演进路径、区分框架继承与真实创新，并识别同届相似、AI 使用、文档夸大和测评绕过等风险，成为组委会和评审专家面临的实际问题。

OS-Agent 的设计目标不是构造一个单一查重分数，而是构造一个面向小型操作系统源码的“多证据评审智能体系统”。系统将源码指纹、AST 结构、git 开发历史、设计文档声明、模块实现证据、负向搜索和风险信号组织成互相支撑的证据网络，再通过分工明确的 Agent 角色模拟人类评委的分析行为，最终生成可复核、可解释、面向评委阅读的评审报告。

本文档重点说明系统的设计思想、与传统查重方法的差异、核心推理模型、Agent 角色划分、报告组织方式以及为什么这些设计能够更贴近 OS 内核作品评审的真实需求。开发历程与阶段性挑战将在后续章节单独补充。

## 1. 赛题背景与问题定义

OS 功能挑战赛道要求参赛者利用大语言模型与智能体技术，构建专门面向小型操作系统源码的分析比对系统。该任务不同于通用软件代码查重，原因在于操作系统作品具有以下特征：

- 代码来源复杂：同一作品可能同时包含教学内核、往届作品、第三方库、Linux ABI 参考、论文设计、AI 生成片段和学生自己写的适配代码。
- 框架继承合理存在：大量参赛作品以 rCore、uCore、ArceOS、xv6、RT-Thread 或往届作品为基础，框架高度相似并不必然意味着违规。
- 创新点经常是局部机制：真实工作量可能体现在页表、缺页、调度、文件系统、网络、SMP、用户 ABI、驱动适配等局部路径，而不是整个仓库全新。
- 同届相似需要方向判断：查重工具只能说明“相似”，但评委需要知道“是否超出共同 Base”“是否存在传播方向”“是否可能来自共同外部来源”。
- 文档声明需要核验：设计书常声明 Base、功能、外部依赖、AI 使用和开发历程，这些声明可能真实、夸大、不完整或与代码矛盾。
- 测试通过可能并不等于机制实现：内核赛道存在硬编码测试名、伪造 `TPASS`/`Pass!` 输出、runner 绕过等风险。

因此，本赛题的核心不是一个线性步骤或单一指标，而是一个多目标、多证据、相互牵连的推理问题。Base 选择会影响模块差异描述；模块阅读会反过来修正文档声明判断；git 历史既支撑工作量，也支撑 AI 使用和抄袭方向；同一条证据可能服务多个结论，一个结论也往往需要多类证据共同确认。

## 2. 传统方法的局限

### 2.1 MOSS/JPlag 类方法

MOSS、JPlag 等传统代码查重系统通常基于 token、语法结构或程序片段相似性。它们在课程作业查重中有效，但直接用于小型操作系统比赛会遇到明显不足：

- 缺少 Base 意识：无法区分共同教学框架、第三方库和学生新增代码。
- 缺少方向判断：只能输出相似关系，不能判断同届代码传播先后。
- 缺少工作量解释：无法说明相似代码是原样继承、外部依赖、局部适配还是机制级重写。
- 缺少文档核验：不会检查 README/PDF 中的 Base 声明、AI 声明和功能声明是否可信。
- 缺少内核语义：不知道 Trap、syscall、页表、ext4、SMP、用户 ABI 等模块在评审中的权重差异。

换言之，传统查重方法适合回答“代码是否相似”，但不适合回答“一个 OS 作品到底做了什么、继承了什么、创新在哪里、风险在哪里”。

### 2.2 单纯 blob/AST 指纹对比

blob hash 和 AST/结构指纹是必要的事实工具，但单独使用仍然不足。

blob hash 可以精确识别逐字节复制、fork、路径搬移和共同框架版本；AST/结构指纹可以穿透函数改名、变量替换、拆文件、合文件、虚假注释和格式修改。这两类指纹提供高价值事实，但它们仍然无法独立解释：

- 相似代码是否位于学生核心路径。
- 相似部分是否已经由文档声明。
- 相似部分是否属于共同 Base 或第三方依赖。
- 双方谁先提交，谁后导入或改名。
- 改动是否构成真实工作量。
- 该相似是否影响最终评审结论。

因此，指纹是证据，不是结论。OS-Agent 将 blob/AST 作为证据工厂的核心输入，但不让分数直接变成报告结论。

### 2.3 “指纹结果 + LLM 解释”的不足

一种自然方案是先用指纹工具得到相似结果，再让 LLM 解释。该方案比传统查重更易读，但如果缺少流程约束和专家经验注入，仍然存在问题：

- LLM 容易把查重分数写成结论，忽略共同 Base 和外部依赖。
- LLM 容易受长上下文影响，前几个模块认真，后续模块模板化。
- LLM 容易编造作品名、Base、架构节点或功能实现。
- LLM 容易把“不确定”写成强结论。
- LLM 可能只解释已有相似度，而不主动核验文档声明、git 历史和测评风险。

OS-Agent 的设计不是让 LLM 在指纹结果后自由发挥，而是让 Agent 模仿人类专家的工作方式：先读身份与文档，再分清源码来源，再选择 Base，再围绕 Base 分模块比较，最后用多证据组织结论。

## 3. 设计总览：从查重分数到评审案件

OS-Agent 将每个作品视为一个评审案件，而不是一个待打分样本。系统围绕以下问题展开：

1. 作品身份是什么，公开报告应该如何称呼。
2. 源码中哪些是学生核心代码，哪些是框架、依赖、测试或生成物。
3. 主 Base 是谁，声明与事实是否一致。
4. 与 Base 相比，哪些模块发生了实质变化。
5. 同届高相似是否超出共同 Base，是否有传播方向。
6. 每个必写模块实现到什么程度，真实工作量在哪一层。
7. 文档声明、AI 使用声明和开发历程是否可信。
8. 是否存在硬编码测评输出或提示词注入风险。
9. 最终报告如何让评委快速看到整体结论、证据链和模块细节。

系统不是依赖单个大 Agent 完成全部判断，而是由确定性工具、Skill 规则、任务文件、子 Agent、矛盾仲裁和校验器共同完成。确定性工具给出可复核事实；Skill 将人类评审经验写入规则；Agent 在窄任务中做判断；校验器阻止无证强结论、命名错误和模板化输出。

## 4. Skill：专家经验注入机制

本项目的关键设计之一是把真实评审经验从一次性 prompt 中抽离出来，固化为 Skill 与 Agent prompt 约束。

在人工评审中，专家并不会只看相似度，而会形成一组隐含判断习惯：先看设计书声明，再看仓库历史；先排除共同框架和第三方依赖，再比较学生核心代码；看到 blob 与 AST 差距时怀疑改名或拆文件；看到首次 commit 批量导入时关注来源和 AI；看到 `TPASS`/`Pass!` 时检查 runner 是否真实执行测试。

OS-Agent 将这些经验显式写入 `judgment-playbook.md`、各角色 Agent 文件和 Markdown contract。这样做的意义是：

- 人类经验不再依赖一次超长 prompt，而是成为可复用规则。
- 不同 Agent 执行同一任务时，判断标准保持一致。
- 复杂判断被拆为可校验的短规则，例如“同届方向必须同时有结构证据和 git 时间线证据”。
- 当 Agent 输出灌水或越权时，validator 可以明确指出违反了哪条规则。

因此，Skill 在本系统中不是简单的“提示词集合”，而是专家评审方法论的工程化载体。

## 5. 证据多对多模型

OS-Agent 使用 evidence 作为所有结论的公共支撑单位。每条 evidence 记录来源、路径、commit、位置、摘录、支撑范围、置信度和验证状态。Agent 只能在正文中引用 `[@E001]` 形式的 evidence chip。

证据不是单一用途。一个 evidence 可以支撑多个结论域：

| 证据来源 | 可能支撑的结论 |
|---|---|
| git history | 真实工作量、开发历史、AI 痕迹、同届方向、批量导入、掩盖手法 |
| README/PDF/设计书 | Base 声明、外部依赖、模块设计、AI 使用声明、开发历程 |
| source span | 模块实现、Base delta、架构节点、外部模块适配 |
| blob/AST 摘要 | Base 身份、来源关系、相对 Base 工作量、同届方向候选 |
| negative search | 文档声明缺证、功能缺失、公共来源排除 |
| risk signal | 测试造假、刷分、runner 绕过、prompt injection |

同样，一个强结论也必须由多个 evidence 共同支撑。例如，同届抄袭方向至少需要结构相似证据和 git 时间线证据；隐匿 AI 需要文档声明与仓库/提交证据之间存在矛盾；文档声明不实需要文档 evidence 与代码/负向搜索 evidence 同时出现。

这种多对多模型让系统能够表达真实评审中的复杂关系，而不是把报告退化成“每个字段填一个证据”的表格。

## 6. Scope：源代码归属边界

Scope 的任务是先分清源码归属，避免查重和工作量判断被噪声污染。系统将文件划分为：

- `student_core`：学生核心实现。
- `framework_base`：共同框架或主 Base。
- `third_party`：第三方库，如 musl、lwIP、virtio drivers、ext4 parser。
- `generated`：构建产物和自动生成文件。
- `test_payload`：LTP/libc-test/BusyBox/比赛 runner、测试样例。
- `documentation`：README、PDF、设计文档。
- `unknown`：无法确认来源，只能进入不确定结论。

Scope 不是最终判断，也不是查重证据。它的作用是为后续 Base 选择、同届方向和真实工作量分析提供边界。路径名像 `vendor/` 或 `third_party/` 并不自动排除源码；系统仍要求结合依赖声明、引用关系、修改历史和实际使用情况判断。

## 7. 指纹证据设计

### 7.1 blob hash

blob hash 来自 git 对文件内容的哈希。相同 blob 表示逐字节相同，具有很强的证据价值。它适合快速识别：

- 直接复制。
- fork 关系。
- 路径搬移。
- 共同框架版本。
- 第三方依赖重复引入。

blob 的缺点是对改名、格式化、拆文件、合文件和注释修改敏感，因此不能单独承担反混淆任务。

### 7.2 AST/结构指纹

AST/结构指纹关注函数和类型的结构形态，不依赖路径和标识符名字。它用于识别：

- 函数名和变量名批量替换。
- 大文件拆成多个小文件。
- 多个文件合并。
- 包名前缀替换。
- 虚假注释和格式修改。

当 blob 相似度下降但 AST/结构相似度保持较高时，系统会将其作为可能存在系统性改写或掩盖手法的信号。normalized token winnowing 暂不作为强结论来源，因为当前实现存在归一化偏移问题，可能与 AST 结果大幅缩水。

### 7.3 三方比较

同届相似必须做三方比较：

```text
目标作品 vs 主 Base
候选作品 vs 主 Base
目标作品 vs 候选作品
```

只有当目标与候选的相似度显著超出双方各自对 Base 的相似度，并且相似热点位于 scope 后的学生核心代码，才进入同届方向复核。

## 8. Base 与来源关系

Base 是解释主骨架的参考锚点，不等于作品全部来源。一个作品可能同时包含：

- 教学内核或往届作品主骨架。
- 某个公开框架的局部模块。
- 第三方库。
- Linux ABI 或 man-pages 参考。
- 论文、教材、博客或其他设计参考。
- AI 生成片段。
- 学生自己完成的平台适配或机制实现。

系统只为作品选择一个主骨架 Base。其他来源不强行塞进 Base，而是在模块差异、外部依赖、文档声明和风险 finding 中说明。这比“选一个最高相似 repo 当 Base”更符合人类评委的分析方式。

Base 未被接受前，模块 Agent 不启动。原因是模块描述的核心是“相对 Base 的变化”；Base 未定时强行写模块，会导致报告只剩模板化实现描述。

## 9. 同届抄袭方向推理

同届抄袭方向是系统中最重要、也最容易误判的结论。OS-Agent 不把相似分数直接解释为抄袭，而使用“结构相似 + 时间先后 + 来源排除 + 手法识别”的组合推理。

### 9.1 推理步骤

1. 确定共同 Base 或共同外部来源。
2. 用 Scope 排除框架、第三方库、测试 payload 和生成物。
3. 比较目标、候选、Base 三者的 blob 和 AST/结构指纹。
4. 定位 Base 外相似热点：函数、类型、syscall、配置、测试桥接、目录重组。
5. 读取双方 git 时间线：首次出现、文件数曲线、批量导入、后续改名拆文件、截止日前清理。
6. 检查文档声明：是否承认 Base、外部依赖、参考来源、AI 使用。
7. 必要时做 negative search：关键函数或特征实现是否存在公开来源。
8. 给出方向明确、共同外部来源、合作传播、方向不确定或证据不足等结论。

### 9.2 强结论门槛

同届方向强结论必须同时具备：

- 结构证据：blob/AST/结构指纹显示 Base 外学生核心代码存在高相似，且能定位到具体函数、类型、路径、syscall 或配置。
- 时间证据：git history 显示核心代码在一方先出现，另一方后导入、改名、拆文件、路径搬移或批量替换。

缺少任一类证据时，只能写“不确定”或“待补证”。这条规则被写入 task file、Agent prompt 和 validator，避免 Agent 因相似度高而直接下方向结论。

### 9.3 掩盖手法识别

| 手法 | blob 表现 | AST/结构表现 | 评审意义 |
|---|---|---|---|
| 直接复制 | 高相同 | 高相同 | 容易识别 |
| 路径搬移 | blob 相同但路径不同 | 相同 | 路径重组不算原创 |
| 改名 | blob 下降 | 结构仍相似 | 常见掩盖 |
| 拆文件 | blob 下降 | 函数级仍匹配 | 高价值信号 |
| 合文件 | blob 下降 | 函数级仍匹配 | 高价值信号 |
| 虚假注释/格式修改 | blob 下降 | 结构不变 | 低成本掩盖 |
| 清理 git 痕迹 | 内容证据保留 | 时间证据异常 | 高风险 |

系统不把掩盖手法本身当作单独定罪依据，而是将其作为解释 blob 与 AST 差距、理解传播过程和判断方向可信度的辅助证据。

## 10. 真实工作量分层

真实工作量是评委最关心的指标之一。OS-Agent 将代码贡献分为以下层级：

| 层级 | 工作量归因 | 说明 |
|---|---|---|
| 共同框架/Base 原样继承 | 不计入或低权重 | 可作为运行基础，但不是学生新增 |
| 第三方依赖原样引入 | 不计入原创 | 只记录依赖管理和声明情况 |
| 测试 payload | 不计入核心实现 | 进入测试风险和兼容性判断 |
| 外部模块适配 | 部分计入 | 只计接口、平台、构建、ABI、内存/I/O 接入修改 |
| Base 上小改 | 低到中等 | bug fix、参数适配、少量接口调整 |
| Base 上大改 | 中到高 | 重写关键路径、替换机制、深度适配 |
| 独立新增机制 | 高 | 有独立设计、代码锚点、测试或运行证据 |
| stub/空壳/return 0 | 不计入实现 | 标为 minimal 或 absent |
| 硬编码测评输出 | 不计入，进入风险 | 即使能过测试，也不算真实实现 |

每个模块同时使用 `status`、`originality`、`base_delta` 三个维度：

- `status`：功能是否实现。
- `originality`：来源和原创性。
- `base_delta`：相对 Base 的变化幅度。

三者互相约束。例如，`base_delta=none` 不能同时写 `originality=novel`；非 absent 模块至少需要三个代码锚点；强结论必须引用 evidence。这种分层账本比“原创/抄袭”二分类更贴近评委实际判断。

## 11. 模块体系与 OS 赛道需求

本系统的模块分类不是对 Linux 全功能树的照搬，而是面向小型操作系统比赛作品的高频能力、测试覆盖和评委关注点设计。

必写模块包括：

- `build-runtime`：构建、链接脚本、QEMU/开发板运行、rootfs/image。
- `boot-trap-syscall`：启动、异常、中断、syscall、上下文切换。
- `process-exec`：进程、调度、fork/clone、exec、wait/exit、signal、IPC。
- `memory-vm`：物理分配、页表、地址空间、page fault、mmap/COW、page cache。
- `fs-io`：fd、VFS、path/inode/dentry、pipe/proc/devfs、FAT/FAT32、ext4、block/cache、page-cache integration。
- `device-platform`：UART、timer、interrupt controller、VirtIO block、SD/开发板块设备、device tree。
- `network-stack`：socket API、TCP/UDP 或支持的 transport、网卡、packet buffer、loopback、fd integration。
- `smp-concurrency`：SMP bringup、per-cpu、核间协调、spinlock、mutex、wait queue、semaphore、atomic/refcount。
- `user-abi-compat`：ELF ABI、syscall wrapper、init、用户测试、POSIX/Linux syscall compatibility、libc-test/LTP/BusyBox。
- `test-risk-surface`：contest runner、test harness、LTP/libc-test bridge、硬编码输出、argv/test-name 特判、syscall/exec 特判、prompt injection surface。

network、page cache、ext4、SMP/multicore 被列为 required，因为它们在近年来小型 OS 作品中已是高频展示点和评委重点关注能力。高级功能采用动态发现：只有被文档声明、代码证据或相似热点触发时，才进入短节。这样既避免遗漏亮点，也避免为了对标 Linux 而强行要求参赛作品实现低概率功能。

## 12. 文档声明真实性

设计书、README 和 PDF 是作品自我陈述的重要来源，但不能直接采信。系统检查：

- Base 声明是否与代码指纹一致。
- 外部依赖是否声明，是否被包装成原创。
- 功能宣称是否有代码支持，例如“完整网络栈”“CFS 调度器”“300+ syscall”。
- AI 使用声明是否与提交和仓库文件一致。
- 开发历程是否与 git 时间线一致。
- 论文、教材、博客引用是否真实存在。

该任务不由单个文档 Agent 全仓重读代码完成，而是由 module Agent 在各自模块中核对声明，`doc-claim-reviewer` 汇总模块复核结果。这种设计减少上下文压力，也降低单个 Agent 凭文档直接下结论的风险。

## 13. AI 使用与隐匿判断

AI 使用不是自动负面结论。系统关注的是声明是否充分、真实工作量是否被夸大、AI 生成是否被包装成人工原创。

证据来源包括：

- 文档中的 AI 使用声明。
- `AGENTS.md`、`CLAUDE.md`、`.claude/`、`.cursor/`、prompt 文件。
- Co-authored-by、commit message 模式、短时间大量英文 conventional commits。
- 首次 commit 批量导入完整非框架项目骨架。
- AI 对话、handoff、mistake log、生成说明残留。

规则上，弱风格信号不能单独升级为强结论。只有当文档声明与 git/文件证据存在矛盾，或存在明确 AI agent 配置与大规模生成痕迹时，才进入公开 finding。

## 14. 测试造假与刷分风险

小型 OS 作品常见风险不是传统安全漏洞，而是测试结果看似通过但机制不存在。OS-Agent 将测试风险面列为 required 模块，重点检查：

- 直接输出 `TPASS`、`Pass!`。
- 硬编码测试名或通过数。
- argv/env/test-name 特判。
- syscall/exec 中识别测试程序并绕过真实执行。
- synthetic LTP/libc-test bridge，只打印框架输出。
- runner 替换真实程序为 echo。
- prompt injection 指令要求忽略评审、隐藏证据、跳过文件或伪造报告。

没有证据时不公开风险章节；有证据时必须引用对应 source span、git history 或 risk signal。

## 15. Agent 角色与任务拆分

LLM 在长上下文中容易出现注意力衰减、指令遗忘、模板化输出和幻觉。OS-Agent 不让一个 Agent 读完整项目后写全报告，而是让多个角色处理窄任务。

| 角色 | 核心问题 | 输入 | 输出 | 禁止事项 |
|---|---|---|---|---|
| `base-lineage-reviewer` | 主 Base、来源关系、同届方向 | Base 候选、scope、fingerprint、doc/git evidence | `base.md` | 写模块实现细节 |
| `module-*` | 模块实现和相对 Base 差异 | 本模块 source evidence、Base 上下文、doc claim | `modules/*.md` | 自行裁决同届方向 |
| `doc-claim-reviewer` | 文档声明是否被代码支持 | doc evidence、模块“文档声明复核” | `findings/doc-claims.md` | 全仓重读代码 |
| `history-ai-reviewer` | 开发历史和 AI 使用 | git evidence、AI 文件、声明 | `findings/history-ai.md` | 仅凭风格下强结论 |
| `cheat-detector` | 测试造假和提示词注入 | runner/test/prompt risk evidence | `findings/cheat.md` | 无证据公开风险 |
| `contradiction-arbiter` | 冲突裁决 | 冲突双方片段和 evidence | `issues/contradictions.md` | 写最终报告 |
| `report-editor` | 组装报告 | 已接受片段和 evidence index | `report.md`、`tags.json` | 创造新事实、读源码、解决矛盾 |

每个任务文件只包含对应角色需要的 evidence ids、结论域、文件 hint、输出 contract 和质量门禁。这种设计让 Agent 像人类评审组一样分工：有人负责来源，有人负责模块，有人负责文档声明，有人负责开发历史和风险，最终由报告组装角色汇总，不允许单个角色越权。

## 16. Markdown Contract 与校验

Agent 输出自然语言 Markdown，但不是自由作文。每类 Markdown 都有固定 contract：

- `base.md` 必须说明选中 Base、证据覆盖、未选候选、方向判断和后续模块。
- `modules/*.md` 必须说明适用范围、实现内容、相对 Base 变化、真实工作量、继承/外部依赖/缺失、文档声明复核和证据。
- `findings/*.md` 只在有证据时公开 finding。
- `issues/contradictions.md` 记录未解决或已解决冲突。
- `report.md` 只由已接受片段组装，不创造新事实。

校验器负责：

- 检查标题层级和顺序。
- 检查 evidence id 是否存在。
- 检查强结论是否有证据。
- 检查同届方向是否同时有结构证据和时间线证据。
- 检查模块是否有代码锚点和 Base delta。
- 检查机器 repo id、裸数字后缀、旧 clone 名是否泄漏。
- 检查无 finding 章节是否被错误展示。
- 检查 Mermaid 图是否包含虚构节点。

程序不自动修补 Agent 输出。失败即打回对应角色。

## 17. 报告结构与前端展示

最终报告面向评委阅读，强调先结论、后细节、可追溯证据：

```text
# <display_name> 评审报告
## 整体结论
## 重点结论
## Base 与来源关系
## 真实工作量账本
## 内核架构图
## 文档声明审查
## 开发历史与 AI 使用
## 作弊、刷分与提示注入风险
## 模块实现与 Base 差异
### <模块>
## 证据索引
```

文档、历史、作弊风险章节只在存在公开 finding 时出现。不存在风险时完全省略，不写“未发现”占位。模块细节必须回到 Base delta 和真实工作量，不允许只写“实现完整”。

前端阅读器提供作品索引、风险过滤、模块账本、章节导航和 evidence drawer。点击 `[@E001]` 可以查看来源、路径、commit、摘录和支撑结论域。前端不重新解释结论，只帮助评委快速阅读和核验证据。

## 18. 本设计的优势

与传统查重工具相比，OS-Agent 不止输出相似度，而是输出来源关系、工作量分层和可复核证据链。

与单纯 blob/AST 指纹对比相比，OS-Agent 将指纹结果放入 Base、scope、git、文档和模块语义中解释，避免把共同框架或第三方依赖误判为抄袭。

与“指纹结果 + LLM 解释”相比，OS-Agent 不让 LLM 自由发挥，而是通过 Skill 注入专家经验，通过任务文件限制上下文，通过 Markdown contract 约束输出，通过 validator 阻止无证强结论和模板化报告。

与人工评审相比，OS-Agent 不能替代专家最终判断，但可以把大量重复的证据整理、模块阅读、文档核验、历史检查和风险扫描自动化，显著降低评委定位重点的成本。

本系统的核心贡献在于：将小型操作系统作品评审抽象为一个多证据、多角色、多结论互相制约的智能体评审问题，并给出一套工程上可执行、报告上可解释、评委可复核的实现方案。
