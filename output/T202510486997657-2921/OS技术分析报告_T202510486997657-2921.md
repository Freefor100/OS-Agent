# 第01章 项目概览与技术栈

## 第 1 章：项目概览与技术栈

## 快速总览

**F7LY-OS** 基于 **Xv6** 修改的 **RISC-V/LoongArch 双架构** 宏内核，主要语言 **C++23**，**自研网络协议栈 (onpstack) 与 lwext4 集成** 为最突出技术点。

## 评测与交付适配

- **Delivery**：`Makefile` 定义 `all` 目标同时构建双架构 (`Makefile:24`)，产物为 `build/riscv/kernel-qemu` 与 `build/loongarch/kernel-la`（通过 `OUTPUT_PREFIX` 区分）。块设备镜像为 `sdcard-rv-onsite.img` 与 `sdcard-la-onsite.img` (`Makefile:300-315`)，适配现场赛评测环境。
- **Harness**：`ltp_judge/` 目录含 LTP 测试判定脚本 (`judge_ltp_musl.py:99 行 `、`analyze_ltp_results.py:159 行`)，`ltp_scroeboard/` 含成绩单解析工具。README 声称"210 余个系统调用通过 LTP 测试"，但代码实际实现约 180 个（第 02 章 Q02_027）。
- **PlatformProfile**：QEMU 启动参数为 `-machine virt -smp 1` (`Makefile:28-29`)，使用 QEMU `virt` 虚拟平台，非物理板（如 VisionFive2）。双架构通过 `ARCH` 变量互斥选择 (`Makefile:14-23`)，代码中通过 `#ifdef RISCV` / `#ifdef LOONGARCH` 区分 (`kernel/platform.hh:6`)。
- **SubsystemDepth**：README 声称支持 FAT32/ext4 双文件系统，但 `kernel/fs/fat32/` 仅 1,353 行（第 10 章 10.3.3），功能完整性待 LTP 验证。网络协议栈为自研 onpstack（`kernel/net/onpstack/` 约 2,080 行），已实现 socket/bind/connect/sendto/recvfrom（第 08 章 Q08_003）。

## 各模块技术全景（基于 02–10 章报告提取）

### 02 启动/架构与 Trap/系统调用

##### 技术清单
- 启动链与引导交接：固件/引导加载器 → 内核入口（OpenSBI/SBI 交接到 `_entry` 汇编入口）
- 特权级与执行模式（硬件隔离模型）：RISC-V M→S 模式切换，LoongArch PLV0/PLV3 隔离
- MMU 与内核地址空间初建：启动早期建立内核页表，启用 satp/pgdl
- 同步异常与用户态陷阱入口（含 syscall 路径）：ecall/syscall 指令触发 usertrap，通过 syscall 号分发表调用
- 异步设备中断与中断控制器抽象：PLIC（RISC-V）/ExtIOI（LoongArch）中断分发
- 时钟源与定时中断（tick/计账/抢占触发）：CLINT/TCFG 定时器触发时钟中断，调用 scheduler.yield()
- 用户内存访问与系统调用参数安全（copyin/out 等）：通过页表 walk 验证用户地址，copy_in/copy_out 实现安全拷贝

##### 关键实现、证据与细粒度锚点
- RISC-V 入口：`kernel/boot/riscv/entry.S:3` `_entry` 标签，加载地址 0x80200000（`kernel/link/riscv/kernel.ld:9`）
- LoongArch 入口：`kernel/boot/loongarch/entry.S:4` `_entry` 标签，加载地址 0x9000000080000000（`kernel/link/loongarch/kernel.ld:6`）
- 模式切换：RISC-V 设置 sstatus.SPP/SPIE 位（`kernel/trap/riscv/trap.cc:50-54`），LoongArch 设置 CRMD.PLV/IE 位（`kernel/trap/loongarch/trap.cc:47-57`）
- TrapFrame 定义：`kernel/proc/trapframe.hh:14-42`（RISC-V 37 字段/296 字节），`kernel/proc/trapframe.hh:44-79`（LoongArch 36 字段/288 字节）
- 系统调用分发表：`kernel/sys/syscall_handler.cc:68-370` 通过 `BIND_SYSCALL` 宏注册约 150+ 个 syscall
- 用户指针检查：`kernel/mem/virtual_memory_manager.cc` 中 `copy_in/copy_out` 通过 `pt.walk_addr()` 验证地址

##### 依赖与工具
- 依赖 OpenSBI/SBI 固件（RISC-V）与 LoongArch 固件规范
- 使用 QEMU 9.2.1 模拟 virt 平台，GDB 调试（`gdb-multiarch` / `loongarch64-linux-gnu-gdb`）
- 构建工具：Makefile 条件编译（`-DRISCV` / `-DLOONGARCH`）

##### 与相邻模块的衔接
- 与第 03 章页表切换联动：启动链中 `trap_mgr.init` 后调用 `k_vmm.init()` 建立内核页表（`kernel/boot/riscv/main.cc:54-60`）
- 为第 04 章进程调度提供 TrapFrame 保存/恢复机制，usertrap 返回前调用 `start_schedule()` 触发调度
- 与第 06 章信号处理衔接：trap 返回前检查 `pending_signal`，调用 `handle_signal()` 构建用户态 handler 上下文（第 02 章 Q02_024）

### 03 内存管理

##### 技术清单
- 物理内存组织与页帧分配器：伙伴系统（Buddy System）+ Slab 分配器
- 页表、地址空间与虚实地址转换：Sv39 页表（RISC-V），软件 walk 映射
- 缺页与页面错误处理（含按需分页/惰性路径）：缺页异常触发 `mmap_handler`，调用 `allocate_vma_page` 懒分配
- 进程虚拟地址空间布局与映射接口：VMA 数组管理 mmap 区域，heap/stack 通过 `uvmalloc/uvmfirst` 映射
- 高级策略（CoW/Lazy/换页/mmap 等）：实现 Lazy Allocation 与 mmap（文件/匿名映射），未实现 CoW 与 swap
- 页缓存或与 FS 块缓存的边界（归入本章或与第 05 章交叉说明）：未实现 Page Cache，依赖 FS 层 bcache

##### 关键实现、证据与细粒度锚点
- BuddySystem：`kernel/mem/buddysystem.cc:146` `Alloc()` 使用完全二叉树（`uint8* tree`）标记节点状态（`NODE_UNUSED/NODE_USED/NODE_SPLIT/NODE_FULL`）
- 页表操作：`kernel/mem/riscv/pagetable.cc:82-147` `PageTable::walk()` 软件遍历页表，`kernel/mem/virtual_memory_manager.cc:93-138` `map_pages()` 建立映射
- 缺页链路：`kernel/trap/riscv/trap.cc:226` `usertrap()` → `mmap_handler()` → `allocate_vma_page()` → `k_pmm.alloc_page()` → `map_pages()`（第 03 章 Q03_011）
- VMA 结构：`kernel/proc/proc.hh` 中 `vma _vm[NVMA]` 数组（NVMA=50），记录 `addr/len/prot/flags/vfile/offset`
- mmap 实现：`kernel/sys/syscall_handler.cc:2170-2253` 完整实现，调用 `proc::k_pm.mmap()`
- TLB 刷新：RISC-V `sfence_vma()`（`kernel/platform.hh:349`），LoongArch `invtlb` 指令（`kernel/mem/virtual_memory_manager.cc:32`）

##### 依赖与工具
- 无外部 crate/库依赖，仅标准工具链与构建系统
- 依赖硬件 MMU（Sv39 页表机制）

##### 与相邻模块的衔接
- 与第 02 章缺页异常联动：trap 中检测 `scause=12/13/15` 调用 `mmap_handler()`（第 02 章 Q02_025）
- 为第 04 章进程地址空间复制提供支持：`fork()` 中调用 `parent_mm->clone_for_fork()` 深拷贝内存管理器（第 04 章 Q04_008）
- 与第 05 章块缓存边界：PMM 分配物理页给 bcache，但无统一 Page Cache 管理（第 05 章 Q05_008）

### 04 进程/调度与多核

##### 技术清单
- 进程或线程抽象与调度实体（PCB/TCB）：统一 `Pcb` 结构，通过 `_tgid/_tid` 区分进程与线程
- 调度策略与就绪队列结构：优先级调度（Priority），遍历进程池选择 `_priority` 最小（最高优先级）的 RUNNABLE 进程
- 抢占模型与时间片/优先级（可协作则注明）：完全抢占（Fully Preemptive），时钟中断触发 `scheduler.yield()`
- 上下文切换与内核栈/寄存器约定：`swtch.S` 保存/恢复 callee-saved 寄存器（RISC-V s0-s11/ra/sp，LoongArch s0-s8/ra/sp/fp）
- 生命周期（创建/执行/阻塞/退出/wait 与僵尸）：UNUSED→USED→RUNNABLE→RUNNING→ZOMBIE→UNUSED 状态流转
- 多核、每 CPU 状态与 IPI/迁移（若适用）：单核系统（NUMCPU=1），未实现 IPI/任务迁移

##### 关键实现、证据与细粒度锚点
- Pcb 结构：`kernel/proc/proc.hh:76` 定义 `class Pcb`，含 `_context`（Context）、`_state`（State）、`_pid`（PID）、`_trapframe`（TrapFrame）字段
- 调度器入口：`kernel/proc/scheduler.cc:43` `start_schedule()` 主调度循环，`kernel/proc/scheduler.cc:28-38` `get_highest_proirity()` 选择最高优先级进程
- 上下文切换汇编：`kernel/proc/riscv/swtch.S:4-37` 保存/恢复 14 个寄存器（ra/sp/s0-s11）
- fork 实现：`kernel/proc/proc_manager.cc:1025` 设置 `np->_parent = p`，`kernel/proc/proc_manager.cc:1155` 调用 `parent_mm->clone_for_fork()` 复制地址空间
- exit 回收：`kernel/proc/proc_manager.cc:1643` `exit_proc()` 释放地址空间/文件表，设置 ZOMBIE 态并 `wakeup(parent)`
- PID 分配：`kernel/proc/proc_manager.cc:100-105` `alloc_pid()` 使用 `_cur_pid` 自增（只分配不回收）

##### 依赖与工具
- 依赖 EASTL 容器（`eastl::vector` 管理进程池）
- 无外部调度框架，自研优先级调度器

##### 与相邻模块的衔接
- 与第 02 章系统调用衔接：`sys_fork/sys_exec/sys_exit` 直接调用 `proc::k_pm` 接口（第 02 章 Q02_021）
- 与第 06 章同步原语衔接：`sleep/wakeup` 使用 `SpinLock` 保护状态变更，防止丢失唤醒（第 06 章 Q06_004）
- 与第 03 章内存管理衔接：`fork()` 中根据 `CLONE_VM` 标志决定深拷贝或共享内存管理器（第 04 章 Q04_008）

### 05 文件系统与设备 I/O

##### 技术清单
- VFS 与 inode/file 等对象模型：C 风格操作函数指针结构体（`struct file_operations` / `struct inode_operations`）
- 路径解析与挂载/命名空间：实现 `namei/path_walk/lookup`，支持绝对/相对路径与 `.` `..`
- 具体文件系统实现形态：lwext4（第三方 BSD 许可）+ 自研 FAT32（1,353 行）
- 文件描述符与打开文件表：固定数组 `struct ofile {_ofile_ptr[max_open_files]}`（max_open_files=128）
- 块缓存、写回与磁盘 I/O 路径：两层 bcache（内核 bcache + lwext4 ext4_bcache），均采用 LRU 驱逐
- 字符设备与块设备驱动框架（含 virtio 等）：VirtIO 块设备/网卡驱动，MMIO 地址硬编码

##### 关键实现、证据与细粒度锚点
- VFS 操作表：`kernel/fs/vfs/file.hh:10-22` 定义 `struct file_operations`（dup/read/write/close/fstat），`kernel/fs/vfs/inode.hh:35-54` 定义 `struct inode_operations`
- lwext4 集成：`kernel/fs/lwext4/ext4.cc:81.6KB` 实现 `ext4_generic_open2`，`kernel/fs/vfs/vfs_ext4_ext.cc` 封装 `vfs_ext_openat`
- 文件打开路径：`sys_openat` → `vfs_openat` → `vfs_ext_openat` → `ext4_open`（第 05 章 Q05_004）
- bcache LRU：`kernel/fs/bio.cc:165-175` 双向链表头部插入最近使用块，`kernel/fs/lwext4/ext4_bcache.cc:54-60` 红黑树按 `lru_id` 排序
- VirtIO 驱动：`kernel/fs/drivers/riscv/virtio_disk2.cc` MMIO 操作，`kernel/net/drivers/virtio_net.cc` 描述符环 DMA
- 驱动初始化：`kernel/boot/riscv/main.cc:80-110` 顺序：`k_printer.init()` → `virtio_disk_init()` → `init_fs_table()` → `binit()` → `vfs_ext4_init()`

##### 依赖与工具
- lwext4 第三方库（BSD 许可，`kernel/fs/lwext4/` 约 81.6KB）
- 自研 FAT32 驱动（`kernel/fs/fat32/` 1,353 行）
- VirtIO 规范（MMIO/PCI 配置空间访问）

##### 与相邻模块的衔接
- 与第 03 章 mmap 衔接：文件映射通过 VFS `file->mmap()` 接口调用 FS 层实现（第 05 章 Q05_009）
- 与第 08 章网络衔接：socket 作为 VFS 文件对象（`socket_file.cc:28.9KB`），通过 `file_operations` 表统一接口（第 05 章 Q05_014）
- 与第 06 章管道衔接：`pipe_file.cc` 实现环形缓冲区+sleep/wakeup，对接 VFS 文件接口（第 06 章 Q06_006）

### 06 同步与 IPC

##### 技术清单
- 自旋锁与中断上下文临界区规则：`SpinLock` 使用 CAS 原子操作，获取时关中断保护临界区
- 可睡眠互斥与锁序/死锁约束（若述及）：`SleepLock` 组合 `SpinLock+sleep`，未实现全局锁顺序规范
- 等待队列、睡眠与唤醒：`ProcessManager::sleep/wakeup` 使用 `void* _chan` 作为等待通道
- 管道等字节流 IPC：`Pipe` 实现环形缓冲区+sleep/wakeup 阻塞语义
- 信号与异步通知：`signal.cc:42.8KB` 实现 `sigaction/kill/sigreturn`，用户态 handler 通过 `sig_trampoline` 恢复
- 共享内存或 futex 等（若本仓库有）：`Futex` 实现 `futex_wait/futex_wakeup`，`ShmManager` 实现 `shmget/shmat/shmdt`

##### 关键实现、证据与细粒度锚点
- SpinLock：`kernel/devs/spinlock.hh:10-36` 使用 `eastl::atomic<Cpu*>` 实现 CAS，`kernel/devs/spinlock.cc:25-32` `acquire` 关中断
- SleepLock：`kernel/proc/sleeplock.hh:16-30` 内部组合 `SpinLock`，`kernel/proc/sleeplock.cc:15-21` `acquire` 循环检查并调用 `sleep`
- Futex：`kernel/proc/futex.cc:1-278` 实现 `futex_wait/futex_wakeup`，支持用户态快速路径
- Pipe：`kernel/proc/pipe.cc` 环形缓冲区，`read/write` 阻塞时调用 `sleep/wakeup`
- 信号处理：`kernel/proc/signal.cc:824-895` `sig_return()` 从用户栈恢复 `trapframe`，`kernel/mem/riscv/sig_trampoline.S:9` `sig_trampoline` 执行 ecall
- 死锁风险：四条件均可能成立，未发现全局锁顺序规范（第 06 章 Q06_014）

##### 依赖与工具
- EASTL 原子操作库（`eastl::atomic`）
- 无外部 IPC 框架，自研 Pipe/Futex/Signal

##### 与相邻模块的衔接
- 与第 04 章进程调度衔接：`sleep/wakeup` 改变进程状态（RUNNABLE↔SLEEPING），触发调度器重新选择进程（第 06 章 Q06_004）
- 与第 07 章安全机制衔接：`sys_kill` 检查 UID/GID 权限（第 07 章 Q07_012），非特权用户仅能发送信号给自有进程
- 与第 05 章管道衔接：`pipe_file.cc` 使用 `sleep/wakeup` 实现阻塞读写，对接 VFS 文件接口（第 05 章 Q05_013）

### 07 安全机制

##### 技术清单
- 硬件隔离与特权域模型：用户态/内核态隔离（RISC-V S-mode/U-mode，LoongArch PLV0/PLV3）
- 访问控制模型（DAC/MAC/Capability 等，无则写不适用）：自主访问控制 DAC（Unix 权限位：owner/group/other）
- 用户指针验证与内核/用户空间数据拷贝边界：`copy_in/copy_out` 通过页表 walk 验证地址，越界返回 -EFAULT
- 可执行空间保护与权限位策略（W^X 等）：未实现 W^X，页表权限位（R/W/X）由 VMA `prot` 字段控制
- 其他沙箱或策略（seccomp/namespace/cgroup 等，无则写不适用）：`sys_prctl` 桩实现（ENOSYS），未实现 seccomp/namespace/cgroup

##### 关键实现、证据与细粒度锚点
- 特权级配置：RISC-V 设置 `stvec=kernelvec, sstatus|=SSTATUS_SIE`（`kernel/trap/riscv/trap.cc:50-54`），LoongArch 设置 `ecfg/tcfg/eentry`（`kernel/trap/loongarch/trap.cc:47-57`）
- UID/GID 检查：`kernel/sys/syscall_handler.cc:5056-5290` `sys_faccessat` 检查 `current_uid/current_gid` 与文件 `owner_perms/group_perms/other_perms`
- 用户指针验证：`kernel/mem/virtual_memory_manager.cc` `copy_in/copy_out` 调用 `pt.walk_addr(va)`，失败则检查 VMA 范围
- KPTI 实现：RISC-V `trampoline.S` 实现用户态/内核态页表切换（`kernel/mem/riscv/trampoline.S:1-148`），用户态使用独立页表
- 栈保护：`kernel/proc/proc.hh` 中 `Pcb` 结构含 `_kstack` 字段，未显式实现 guard page/canary

##### 依赖与工具
- 依赖硬件特权级（RISC-V S/U-mode，LoongArch PLV0/PLV3）
- 无外部安全框架（如 SELinux/AppArmor）

##### 与相邻模块的衔接
- 与第 02 章系统调用衔接：`sys_setuid/sys_setgid` 检查 EPERM 权限（第 02 章 Q02_021），`sys_faccessat` 检查文件访问权限（第 07 章 Q07_004）
- 与第 03 章内存管理衔接：KPTI 通过 `trampoline.S` 切换 satp 页表，用户态/内核态地址空间隔离（第 07 章 Q07_011）
- 与第 04 章进程管理衔接：`Pcb` 含 `_uid/_gid/_euid/_egid` 字段，`fork()` 复制凭证（第 04 章 Q04_001）

### 08 网络协议栈

##### 技术清单
- 套接字抽象与用户态 API：实现 socket/bind/connect/sendto/recvfrom 系统调用，对接 VFS `socket_file` 对象
- 协议栈分层与数据面实现形态：自研 onpstack 协议栈（Ethernet/ARP/IPv4/ICMP/UDP/TCP/DHCP）
- 网卡驱动与收发包/DMA 路径：VirtIO-Net 驱动，填充描述符环并通知 MMIO_QUEUE_NOTIFY
- 与协议栈缓冲与 sk_buff 类抽象（若适用）：`onps_input` 管理输入缓冲区，未实现 sk_buff 结构

1. sys_setuid/sys_setgid (kernel/sys/syscall_handler.cc:2765-2795):

   - 无搜索权限时返回 SYS_EACCES
