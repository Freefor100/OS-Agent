### Q05_001 VFS 抽象层 (Virtual File System, VFS)接口是什么形态？（Rust trait / C op 表；必须给接口定义证据）

C 风格操作函数指针结构体（C-style operation function pointer structs）。VFS 定义了两个核心操作表：`struct file_operations`（文件操作）和 `struct inode_operations`（索引节点操作），均采用 C 语言函数指针形式，非 Rust trait。证据：`kernel/fs/vfs/file.hh:10-22` 定义 `struct file_operations` 含 `dup`、`read`、`write`、`close`、`fstat` 等函数指针；`kernel/fs/vfs/inode.hh:35-54` 定义 `struct inode_operations` 含 `unlockput`、`read`、`write`、`dirlookup`、`create` 等函数指针。

### Q05_003 若支持 FAT32/Ext4：它是自研还是第三方库/crate？（必须引用 Cargo.toml/Cargo.lock 或 Makefile 引入证据）

Ext4: 第三方库 lwext4（BSD 许可，代码位于 `kernel/fs/lwext4/`，版权头见 `kernel/fs/lwext4/ext4_bcache.cc:1-27`）；FAT32: 自研实现（代码位于 `kernel/fs/fat32/`，无第三方版权标识，直接集成到内核构建系统）。仓库为 C++ 项目，无 Cargo.toml，通过 Makefile 和 include 路径引入 lwext4 源码。

### Q05_004 文件打开路径：文件打开入口（sys_open 或等价）→ VFS 层 → 具体 FS open。列出 3-6 个关键节点并给证据。

文件打开路径关键节点：1. `sys_openat`（`kernel/sys/syscall_handler.cc` 系统调用入口）→ 2. `vfs_openat`（VFS 层统一入口）→ 3. `vfs_ext_openat`（Ext4 扩展路径解析）→ 4. `ext4_open`（lwext4 具体实现）。证据：`kernel/sys/syscall_handler.cc` 中 `sys_openat` 调用 VFS 接口；`kernel/fs/vfs/vfs_ext4_ext.cc` 含 `vfs_ext_openat` 实现；`kernel/fs/lwext4/ext4.cc` 含 `ext4_generic_open2` 底层实现。

### Q05_005 文件描述符表 (File Descriptor Table, FD Table) 的实现形态是什么？（固定数组/Vec/BTreeMap 等；必须给结构体定义证据）

固定数组（Fixed-size array）。每个进程 Pcb 内含 `struct ofile`，其中 `_ofile_ptr[max_open_files]` 为固定长度指针数组（`max_open_files = 128`）。证据：`kernel/proc/proc.hh:57-62` 定义 `struct ofile { fs::file *_ofile_ptr[max_open_files]; ... }`，其中 `max_open_files` 在 `kernel/proc/proc.hh:55` 定义为 `constexpr uint max_open_files = 128;`。

### Q05_007 若存在缓存：驱逐策略是什么（LRU/Clock/FIFO/无驱逐）？必须指出判断依据（字段/算法分支）证据。

LRU（Least Recently Used）。两层缓存均采用 LRU：1. 内核 bcache（`kernel/fs/bio.cc`）：使用双向链表 `bcache.head`，`brelse()` 将释放的块移到链表头部（最近使用），`bget()` 从尾部回收最久未使用的块（`kernel/fs/bio.cc:165-175`）；2. lwext4 ext4_bcache（`kernel/fs/lwext4/ext4_bcache.cc`）：使用红黑树 `lru_root` 按 `lru_id` 排序，`ext4_bcache_find_get()` 递增 `lru_id` 并更新树（`kernel/fs/lwext4/ext4_bcache.cc:54-60`），`ext4_buf_lowest_lru()` 获取最小 `lru_id` 进行驱逐（`kernel/fs/lwext4/ext4_bcache.cc:47`）。

### Q05_023 驱动框架接口是什么？（Rust Driver trait / C driver ops / 注册表；必须引用接口定义证据）

C++ 类与函数指针混合模式。设备驱动通过 VirtualDevice 基类（kernel/devs/device_manager.hh）派生，使用 register_device() 注册到 DeviceManager。块设备通过 BlockDevice 抽象接口（kernel/devs/block_device.hh）定义 read_blocks/write_blocks 纯虚函数。VirtIO 驱动直接使用 MMIO/PCI 寄存器操作函数（kernel/fs/drivers/riscv/virtio_disk2.cc、kernel/fs/drivers/loongarch/virtio_pci.cc），无统一 Driver trait 或 ops 表。

### Q05_024 驱动注册与初始化顺序是什么？（init_drivers/probe/driver_manager 等；列出 3-6 个关键节点并给证据）

驱动初始化顺序（以 RISC-V 为例，kernel/boot/riscv/main.cc:80-110）：1. k_printer.init() → 初始化 console/uart（早期输出）；2. virtio_disk_init() → 初始化 VirtIO 块设备驱动；3. init_fs_table() → 初始化文件系统表；4. binit() → 初始化块缓存；5. fileinit() → 初始化文件表；6. vfs_ext4_init() → 初始化 lwext4 文件系统。LoongArch 类似（kernel/boot/loongarch/main.cc:69-72），额外调用 virtio_probe() 进行 PCI 设备探测。

### Q05_029 MMIO 地址来源是什么？（DTB 提供 / 常量硬编码 / 物理→虚拟转换；必须给证据）

常量硬编码（Hard-coded constants）。VirtIO MMIO 地址在 kernel/platform.hh 中定义为宏常量：RISC-V 的 VIRTIO0/VIRTIO1（kernel/platform.hh:400+），LoongArch 的 PCI 配置空间地址通过 pci_config_address() 计算（kernel/devs/loongarch/pci.cc:100）。DTB 仅用于获取 initrd 等启动参数，不提供设备 MMIO 地址。物理地址通过 to_vir() 转换为虚拟地址（kernel/platform.hh:758）。

### Q05_030 多平台适配是如何通过构建/条件编译选择驱动的？（features/Kconfig/Makefile 规则；必须给证据）

条件编译（#ifdef/#elif defined）。使用 #ifdef RISCV 和 #ifdef LOONGARCH 宏区分架构特定代码。证据：kernel/platform.hh:6 开始使用 #ifdef RISCV；kernel/net/drivers/virtio_net.cc:15-22 使用 #ifdef RISCV / #elif defined(LOONGARCH) 选择 MMIO 或 PCI 初始化路径；kernel/boot/riscv/main.cc 和 kernel/boot/loongarch/main.cc 分别为两架构独立启动代码。Makefile 通过 ARCH 变量选择链接脚本和启动文件（kernel/link/riscv/ vs kernel/link/loongarch/）。
