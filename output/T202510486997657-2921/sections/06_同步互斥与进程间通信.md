### Q06_011 若实现 signal handler：用户态 handler 上下文如何构建？是否存在 sigreturn 恢复原 trap frame？（必须给证据）

用户态 handler 上下文构建：
1. 在 kernel/proc/signal.cc:handle_signal() 中，为待处理信号分配 signal_frame（内核页）
2. 保存当前 trapframe 到 signal_frame->tf
3. 设置用户栈：将 signal_frame 地址、返回标记 guard、has_siginfo 标记压入用户栈
4. 设置 trapframe->epc/era 指向 sig_trampoline（`kernel/mem/riscv/sig_trampoline.S:9`）
5. sig_trampoline 执行 ecall 触发 SYS_rt_sigreturn 系统调用

sigreturn 恢复机制：
- `kernel/proc/signal.cc:824-895` sig_return() 函数从用户栈读取 guard 标记验证
- 读取 has_siginfo 判断是否有 siginfo
- 从 signal_frame 恢复 _sigmask 和 trapframe（memmove 复制）
- 释放 signal_frame 内存
- 返回用户态时 trapframe 已恢复原上下文

### Q06_014 死锁四必要条件（Coffman Conditions）在该内核中是否均成立？
请逐条作答（互斥 Mutual Exclusion / 持有并等待 Hold-and-Wait / 不可剥夺 No Preemption / 循环等待 Circular Wait），并结合 SpinLock/Mutex 的实现给出证据或写「不适用」。

1. 互斥 (Mutual Exclusion)：成立。SpinLock 通过 CAS 原子操作确保同一时刻只有一个 CPU 持有锁（`kernel/devs/spinlock.cc:25-32` acquire 使用 compare_exchange_strong）
2. 持有并等待 (Hold-and-Wait)：成立。SleepLock::acquire 在持有_lock 的情况下循环检查并调用 sleep（`kernel/proc/sleeplock.cc:15-21`），sleep 会释放传入的 lock 但持有进程锁
3. 不可剥夺 (No Preemption)：成立。SpinLock 持有期间不会被强制释放，必须显式调用 release()；SleepLock 必须显式 release 才能唤醒等待者
4. 循环等待 (Circular Wait)：可能成立。代码中未发现全局锁顺序规范（Lock Ordering），存在嵌套锁场景（如 sem_p 先获取 sem->lock 再获取 p->_lock），但未发现 ABBA 死锁检测机制

结论：四条件均可能成立，存在死锁风险
