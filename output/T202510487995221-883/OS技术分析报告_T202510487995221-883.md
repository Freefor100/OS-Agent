## 执行摘要（Executive Summary）

TatlinOS 是一个面向教学与竞赛的操作系统内核项目，基于 2024 年参赛作品 TrustOS 演进开发。项目采用 **Rust 2021** 语言编写（`#![no_std]`），支持 **RISC-V 64** 与 **LoongArch 64** 双架构，通过条件编译实现平台适配。

**技术栈概览**：内核使用 Cargo + Makefile 混合构建，依赖 `buddy_system_allocator`（物理帧分配）、`virtio-drivers`（块设备驱动）、`lwext4_rust`（ext4 文件系统封装）等关键 crate。用户态程序支持 musl/glibc 双运行时环境。

**核心特性**：
1. **双架构统一 HAL**：RISC-V SV39 页表与 LoongArch TLB/DMW 机制的统一抽象
2. **高级内存管理**：写时复制（CoW）、懒分配（Lazy Allocation）、mmap 文件映射
3. **完整进程模型**：支持 fork/clone/exec、FIFO 调度、Futex 同步、信号机制
4. **ext4 文件系统**：通过 lwext4 C 库封装实现完整 VFS 接口
5. **基础 IPC**：管道（Pipe）、共享内存（shm）、信号（Signal）

**实现完成度评估**：核心子系统（内存、进程、中断、文件系统）功能完整，可运行 LTP/LMBench 测试套件及 busybox。但网络功能仅为桩实现（基于 Pipe 模拟 Loopback），调度策略仅支持 FIFO，安全机制不完善（UID 切换无权限检查），且为单核设计（`HART_NUM=1`）。整体完成度约 **70%**，适用于教学实验而非生产环境。

## 结论摘要

1. **项目名称与来源**：本项目名为 **TatlinOS**（仓库名 `T202510487995221-883`），是基于 2024 年参赛作品 **TrustOS** 开发的操作系统内核，参赛队伍为黄羲成、常晔文、罗阳。

2. **技术定位**：这是一个使用 **Rust 2021** 编写的教学/竞赛向操作系统内核，支持 **RISC-V 64** 和 **LoongArch 64** 双架构，基于裸机环境（无标准库，`#![no_std]`）运行。

3. **核心框架**：项目采用模块化设计，包含内存管理（`mm/`）、进程管理（`task/`）、文件系统（`fs/`）、系统调用（`syscall/`）、中断处理（`trap/`）等核心子系统。文件系统基于 **lwext4**（C 语言实现的 ext2/3/4 库）通过 Rust FFI 封装提供 ext4 支持。

4. **构建系统**：使用 **Cargo + Makefile** 混合构建，通过 `make TARGET_ARCH=<arch>` 指定目标架构，支持 QEMU 仿真和真实硬件（"上板"）两种运行模式。

5. **功能实现状态**：已实现基础的分页内存管理（SV39/LoongArch TLB）、进程/线程管理（FIFO 调度）、ext4 文件系统、基础网络 socket 接口（基于 pipe 模拟，无真实网络栈）。部分系统调用为桩函数（如 `sys_sched_setscheduler` 返回 `Ok(0)` 无实际逻辑）。

## 启动入口与链接脚本分析

### RISC-V 架构启动入口

RISC-V 架构的启动入口位于 `os/src/arch/riscv64/qemu/asms/entry.asm`，链接脚本为 `os/linkers/linker-rv.ld`。

**链接脚本关键配置**（`os/linkers/linker-rv.ld`）：

```ld
OUTPUT_ARCH(riscv)
ENTRY(_start)
BASE_ADDRESS = 0xffffffc080200000;

SECTIONS
{
    . = BASE_ADDRESS;
    skernel = .;
    .text : {
        *(.text.entry)
        . = ALIGN(4K);
        strampoline = .;
        *(.text.trampoline);
        ...
    }
}
```

- **ENTRY(_start)**：指定 `_start` 为程序入口点
- **BASE_ADDRESS = 0xffffffc080200000**：内核虚拟地址基址（高位直接映射）
- **.text.entry**：启动代码段，包含 `_start` 符号

**汇编入口**（`os/src/arch/riscv64/qemu/asms/entry.asm:1-24`）：

```asm
.section .text.entry
.globl _start
_start:
    mv tp,a0                          # 保存 hart_id 到 tp
    slli t0, a0, 16                   # t0 = hart_id << 16 (4096 * 16)
    la sp, boot_stack_top
    sub sp, sp, t0                    # sp = stack_top - hart_id * stack_size
    
    # 激活页表（MMU 启用）
    la t0, boot_pagetable
    li t1, 8 << 60                    # SATP 模式：SV39
    srli t0, t0, 12
    or t0, t0, t1
    csrw satp, t0
    sfence.vma
    
    call trampoline                   # 跳转到 trampoline
```

关键操作：
1. **hart_id 获取**：从 `a0` 寄存器获取（由 SBI/U-Boot 传递）
2. **栈指针设置**：每个 hart 分配独立栈空间（4KB × hart_id 偏移）
3. **MMU 早期启用**：在跳转到 `trampoline` 前激活页表（`satp` 寄存器）
4. **页表基址**：`boot_pagetable` 定义在 `.data` 段，包含物理地址到虚拟地址的映射

- **BASE_ADDRESS = 0x9000000090000000**：内核虚拟地址基址（使用 DMW 直接映射窗口）

**汇编入口**（`os/src/arch/loongarch64/qemu/asms/entry.asm:1-20`）：

```asm
.section .text.entry
.globl _start
_start:
    invtlb      0x0, $zero, $zero     # 清空 TLB
    
    # 初始化栈指针
    la          $sp, boot_stack
    li.d        $a0, 4096
    csrrd       $a1, 0x20             # CPUID
    addi.d      $tp, $a1, 0
    addi.d      $a1, $a1, 1
    mul.d       $a0, $a0, $a1
    add.d       $sp, $sp, $a0         # sp = boot_stack + (hart_id+1) * 4096

    bl          init_csr_regs         # 跳转到 CSR 初始化
spin:
    b           spin                  # 其他 hart 自旋等待
```

关键操作：
1. **TLB 清空**：`invtlb` 指令清空 TLB 条目
2. **栈指针设置**：每个 CPU 分配 4KB 栈空间
3. **跳转到 `init_csr_regs`**：进入 Rust 代码进行 CSR 寄存器初始化

## 架构初始化流程（模式切换/FPU/MMU）

### RISC-V 架构初始化

#### 模式切换验证

**关键发现**：RISC-V 架构**未显式执行 M-Mode → S-Mode 切换**。模式切换由 SBI/OpenSBI 固件在跳转到内核前完成。

证据：
- 入口汇编 `entry.asm` 中直接操作 `satp`（S-Mode 页表寄存器）
- `trap_interface.rs` 中操作 `sstatus`、`stvec` 等 S-Mode 寄存器
- 未找到 `mstatus.mpp` 或 `mret` 相关代码

**SBI 固件级启动链**（通过文档和代码推断）：
```
OpenSBI (M-Mode) → U-Boot (S-Mode/H-Mode) → TatlinOS Kernel (S-Mode)
```

#### FPU 初始化

**已启用 FPU**。证据位于 `os/src/arch/riscv64/qemu/trap_interface.rs:77-80`：

```rust
pub fn trap_init() {
    set_kernel_trap_entry();
    // 开启 rustsbi 的浮点指令
    unsafe {
        sstatus::set_fs(FS::Clean);
    }
}
```

- `sstatus::set_fs(FS::Clean)`：设置 `sstatus.fs` 字段为 `Clean`，启用 S-Mode 浮点单元
- 但 `trap.S:52` 中 `.set ENABLE_FPU, 0` 表示**浮点寄存器保存/恢复被禁用**

```asm
# 浮点开关
.set ENABLE_FPU, 0  # 如果设为 0，则不会保存和恢复 fp 寄存器以及 fcsr
```

**结论**：FPU 已启用，但上下文切换时不保存浮点状态（可能导致多任务浮点数据污染）。

#### MMU 启用与页表初始化

MMU 在 `entry.asm` 中早期启用（见前文）。页表由 `boot_pagetable` 定义：

```asm
.section .data
.align 12
boot_pagetable:
    .quad (0x80000 << 10) | 0xcf  # PTE: 物理页 0x80000，权限 VRWXAD
    .zero 8 * 255
    .quad (0x80000 << 10) | 0xcf  # 第二个映射
```

- **映射关系**：
  - `0x0000_0000_8000_0000` → `0x0000_0000_8000_0000`（物理内存直接映射）
  - `0xffff_fc00_8000_0000` → `0x0000_0000_8000_0000`（内核虚拟地址映射）

## 到达内核主函数的路径（完整调用链）

### RISC-V 架构调用链

```
_start (entry.asm:2)
  ↓ csrw satp, 调用 trampoline
trampoline (os/src/main.rs:74-82)
  ↓ add sp, 偏移调整
  ↓ la t0, rust_main
  ↓ jalr zero, 0(t0)
rust_main (os/src/main.rs:116)
  ↓ clear_bss()
  ↓ trap::init()
  ↓ mm::init()
  ↓ task::init()
  ↓ fs::init()
  ↓ task::add_initproc()
  ↓ task::run_tasks()
```

**关键跳转分析**（`os/src/main.rs:74-82`）：

```rust
#[cfg(feature = "riscv64")]
#[no_mangle]
pub fn trampoline(hartid: usize) {
    unsafe {
        asm!("add sp, sp, {}", in(reg) arch::memory_layout::KERNEL_ADDR_OFFSET);
        asm!("la t0, rust_main");
        asm!("add t0, t0, {}", in(reg) arch::memory_layout::KERNEL_ADDR_OFFSET);
        asm!("mv a0, {}", in(reg) hartid);
        asm!("jalr zero, 0(t0)");
    }
}
```

- **栈指针调整**：`sp += KERNEL_ADDR_OFFSET`（从物理地址切换到虚拟地址）
- **参数传递**：`hartid` 通过 `a0` 传递给 `rust_main`

### LoongArch64 架构调用链

```
_start (entry.asm:2)
  ↓ bl init_csr_regs
init_csr_regs (os/src/arch/loongarch64/qemu/cpu.rs:52)
  ↓ 配置 DMW/CRMD/PRMD 等 CSR
  ↓ 设置异常入口 (eentry, tlbrentry)
  ↓ 开启 FPU (euen::set_fpe)
  ↓ rust_main(hart_id())
rust_main (os/src/main.rs:116)
  ↓ 同 RISC-V 流程
```

**关键函数**（`os/src/arch/loongarch64/qemu/cpu.rs:52-143`）：

```rust
#[no_mangle]
pub fn init_csr_regs() {
    // DMW 配置
    dmw0::set_plv0(true);
    dmw1::set_plv0(true);
    
    // 中断配置
    ticlr::clear_timer_interrupt();
    crmd::set_ie(false);
    
    // 特权级与分页
    crmd::set_plv(CpuMode::Ring0);
    crmd::set_pg(true);
    
    // 异常入口
    set_merrentry(trap_handler as usize);  // 机器异常
    tlbrentry::set_tlbrentry(__tlb_rfill as usize);  // TLB 重填
    set_kernel_trap_entry();  // 其他异常 (eentry)
    
    // FPU 启用
    euen::set_fpe(true);
    
    // 跳转到内核主函数
    rust_main(hart_id());
}
```

### 1. BSS 段清零（`os/src/main.rs:62-69`）

```rust
fn clear_bss() {

        fn sbss();
        fn ebss();
```

- **作用**：清零 `.bss` 段（未初始化全局变量）
- **实现**：使用链接脚本定义的 `sbss`/`ebss` 符号确定范围

### 2. 多核启动同步（`os/src/main.rs:116-170`）

```rust
static FIRST_HART: AtomicBool = AtomicBool::new(true);
static INIT_FINISHED: AtomicBool = AtomicBool::new(false);

pub fn rust_main(hartid: usize) -> ! {
    if FIRST_HART.load(Ordering::SeqCst) == true {
        FIRST_HART.store(false, Ordering::SeqCst);
        clear_bss();
        // 主核初始化：trap、mm、task、fs...
        INIT_FINISHED.store(true, Ordering::SeqCst);
    } else {
        // 从核等待主核初始化完成
        while !INIT_FINISHED.load(Ordering::SeqCst) {}
        trap::init();
        mm::activate_kernel_space();
    }
    task::run_tasks();
}
```

- **主核**：执行完整初始化流程
- **从核**：自旋等待后仅初始化必要组件（trap、MMU、定时器）

### 3. 串口地址切换（MMU 启用前后）

**RISC-V**：通过 SBI 调用统一处理，无需关心物理/虚拟地址。

**LoongArch64**（`os/src/arch/loongarch64/qemu/console.rs`）：

```rust
use super::memory_layout::UART_ADDR;
static COM1: Mutex<Uart16550> = Mutex::new(Uart16550::new(UART_ADDR));

struct Uart16550 {
    base_address: usize,
}

impl Uart16550 {
    pub fn putchar(&mut self, c: u8) {
        let ptr = self.base_address as *mut u8;
        unsafe {
            // 检查发送保持寄存器空位
            if ptr.add(5).read_volatile() & (1 << 5) != 0 {
                ptr.add(0).write_volatile(c);
            }
        }
    }
}
```

- **MMU 启用前**：`entry.asm` 中 `asm_putchar` 直接访问物理地址 `0x800000001fe20000`
- **MMU 启用后**：通过 DMW 直接映射窗口访问虚拟地址 `UART_ADDR`

### 4. TLB 重填处理（LoongArch64 特有）

**TLB Refill 异常处理**（`os/src/arch/loongarch64/qemu/asms/tlb.S`）：

```asm
.globl __tlb_rfill
.balign 4096
__tlb_rfill:
    csrwr $t0, 0x8B       # 保存 TLBEHI
    csrrd $t0, 0x1B       # 读取 PGDL（页表基址）
    lddir $t0, $t0, 2     # 页表遍历：三级目录
    lddir $t0, $t0, 1     # 页表遍历：二级目录
    ldpte $t0, 0          # 加载偶数页表项到 TLB
    ldpte $t0, 1          # 加载奇数页表项到 TLB
    tlbfill               # 填充 TLB
    ertn                  # 异常返回
```

- **硬件页表遍历**：LoongArch64 支持硬件自动页表遍历（通过 `PWCL/PWCH` 配置）
- **软件补充**：`__tlb_rfill` 处理 TLB Miss 异常，手动填充 TLB

## 启动流程总结

| 阶段 | RISC-V64 | LoongArch64 |
|------|----------|-------------|
| **固件** | OpenSBI → U-Boot | U-Boot（或 QEMU 直接加载） |
| **入口汇编** | `entry.asm:_start` | `entry.asm:_start` |
| **MMU 启用** | `entry.asm` 中设置 `satp` | `init_csr_regs` 中设置 `crmd.pg` |
| **FPU 启用** | `sstatus::set_fs(FS::Clean)` | `euen::set_fpe(true)` |
| **栈初始化** | 每 hart 4KB，`boot_stack_top` 递减 | 每 CPU 4KB，`boot_stack` 递增 |
| **跳转到 Rust** | `trampoline()` → `rust_main()` | `init_csr_regs()` → `rust_main()` |
| **主核初始化** | trap → mm → logger → task → fs | 同左 |
| **从核等待** | 自旋 `INIT_FINISHED` | 同左 |

**关键差异**：
1. **MMU 启用时机**：RISC-V 在汇编中早期启用，LoongArch64 在 Rust 中配置
2. **地址翻译机制**：RISC-V 使用页表（SV39），LoongArch64 使用 DMW+ 页表混合
3. **FPU 上下文**：RISC-V 禁用保存/恢复，LoongArch64 完整保存
4. **TLB 管理**：RISC-V 软件管理（`sfence.vma`），LoongArch64 硬件辅助（`__tlb_rfill`）

# 内存管理物理虚拟分配器

## 第 3 章：内存管理（物理/虚拟/分配器）

### 物理内存管理实现

#### 物理页分配器：PageCache + Buddy System

本仓库采用 **PageCache（页缓存）+ Buddy System（伙伴系统）** 的两级物理内存管理架构。

**1. 全局页缓存 `PageCache`**

位于 `os/src/mm/frame_alloc/page_cache.rs`，实现了一个基于水位的批量分配器：

```rust
pub struct PageCache {
    free_list: Mutex<Vec<PhysPageNum>>, // 空闲页链表
    count: AtomicUsize,                 // 当前缓存页数
}

impl PageCache {
    const HIGH_WATERMARK: usize = 128; // 缓存页数上限
    const LOW_WATERMARK: usize = 32;   // 缓存页数下限
    const REFILL_BATCH: usize = 16;    // 每次补充页数
    const FLUSH_BATCH: usize = 64;     // 每次刷回页数
}
```

- **分配流程** (`alloc`)：优先从 `free_list` 弹出空闲页；若为空，则批量从 `HEAP_ALLOCATOR` 申请 `REFILL_BATCH` (16) 页。
- **释放流程** (`dealloc`)：页回收到 `free_list`；当页数超过 `HIGH_WATERMARK` (128) 时，批量刷回 `FLUSH_BATCH` (64) 页到堆分配器。

**2. 物理页追踪器 `FrameTracker`**

位于 `os/src/mm/frame_alloc/frame_tracker.rs`，采用 RAII 模式管理单页生命周期：

```rust
pub struct FrameTracker {
    pub ppn: PhysPageNum,
}

impl FrameTracker {
    pub fn alloc() -> Option<Arc<FrameTracker>> {
        let ppn = PAGE_CACHE.alloc()?;
        let ret = Some(Arc::new(FrameTracker::new(ppn)));
        ret
    }
}

impl Drop for FrameTracker {
    fn drop(&mut self) {
        PAGE_CACHE.dealloc(self.ppn);
    }
}
```

- **清零初始化**：构造函数中清空物理页内容 (`bytes_array_mut` 逐字节置 0)。
- **引用计数**：返回 `Arc<FrameTracker>`，支持多所有者（如共享内存、COW）。
- **自动回收**：析构时自动调用 `PAGE_CACHE.dealloc`。

**3. 连续页分配 `ContinuousPages`**

位于 `os/src/mm/heap_allocator.rs`，用于分配多页连续内存（如内核栈）：

```rust
pub struct ContinuousPages {
    base: usize,
    page_num: usize,
    layout: Layout,
}

impl ContinuousPages {
    pub fn new(page_num: usize) -> Option<Self> {
        let size = page_num * PAGE_SIZE;
        let layout = Layout::from_size_align(size, PAGE_SIZE)?;
```

```rust
pub fn sys_brk(brk_addr: usize) -> SyscallRet {
    let task = current_task().unwrap();
    let mut task_inner = task.inner_lock();
    let old_addr = task_inner.user_heappoint;
    let grow_size: isize = (brk_addr - old_addr) as isize;

}
```

```rust
    let memory_set = process.get_locked_memory_set();
```

```rust
pub fn sys_mmap(addr: usize, len: usize, prot: u32, flags: u32, fd: usize, off: usize) -> SyscallRet {
    let map_perm: MapPermission = MmapProt::from_bits(prot).unwrap().into();
    let flags = MmapFlags::from_bits(flags)?;
    
    // 处理 MAP_FIXED
    if flags.contains(MmapFlags::MAP_FIXED) && addr == 0 {
        return Err(SysErrNo::EPERM);
    }
    
    // 匿名映射
    if fd == usize::MAX {
        if !flags.contains(MmapFlags::MAP_ANONYMOUS) {
            return Err(SysErrNo::EBADF);
        }
        return Ok(memory_set.mmap(addr, len, map_perm, flags, None, usize::MAX));
    }
    
    // 文件映射
    let file = task_inner.fd_table.get(fd).file()?;
    Ok(memory_set.mmap(addr, len, map_perm, flags, Some(file), off))
}
```

- `MAP_ANONYMOUS` (1<<5)：匿名映射

```rust
pub fn handle_cow_page_fault(&mut self, va: VirtAddr, vma: &mut MapArea) -> bool {
    let pte = self.find_valid_pte(va.floor())?;
    let pte_flags = pte.get_flags();
    
    // 检查 COW 标志
    if !pte_flags.contains(RVPTEFlags::COW) {
        return false;
    }
    
    // 若仅有一个引用，直接设置为可写
    let frame = vma.data_frames.get(&va.into()).unwrap();
    if Arc::strong_count(frame) == 1 {
        pte.set_flags(flags | RVPTEFlags::WD);
        return true;
    }
    
    // 否则：分配新页 + 复制内容
    let src = pte.get_ppn().bytes_array_mut();
    vma.unmap_one(self, va.into());
    vma.map_one(self, va.into());
    let dst = pte.get_ppn().bytes_array_mut();
    dst.copy_from_slice(src);
    
    true
}
```

### 调度算法与策略（代码证据）

本操作系统实现了**基于就绪队列的 FIFO（先进先出）调度策略**，**不支持优先级调度、时间片轮转或 CFS**。

#### 调度器实现

**文件路径**: `os/src/task/manager.rs:10-65`

```rust
pub mod ready_queue {
    static READY_QUEUE: Lazy<Mutex<VecDeque<Weak<TaskControlBlock>>>> =
        Lazy::new(|| Mutex::new(VecDeque::new()));

    pub fn add_task(task: &Arc<TaskControlBlock>) {
        let mut queue = READY_QUEUE.lock();
        if task_in_queue(&queue, task) {
            panic!("add task fail: task already in queue!");
        } else {
            queue.push_back(Arc::downgrade(&task));
        }
    }

    pub fn fetch_task() -> Option<Arc<TaskControlBlock>> {
        loop {
            if let Some(task) = READY_QUEUE.lock().pop_front() {
                if let Some(task_arc) = task.upgrade() {
                    return Some(task_arc);
                }
            } else {
                return None;
            }
        }
    }
}
```

**关键证据**：
1. **数据结构**：使用 `VecDeque`（双端队列）实现就绪队列，`add_task` 从尾部插入，`fetch_task` 从头部取出，符合 FIFO 语义。
2. **无优先级字段**：`TaskControlBlock` 和 `TaskControlBlockInner` 中**不存在** `priority`、`stride`、`sched_class` 等字段。
3. **调度策略系统调用**：`sys_sched_setscheduler` 和 `sys_sched_getscheduler` 均为桩函数（`os/src/syscall/process/others.rs:151-165`），注释明确说明"使用的是标准的时间片调度算法，直接返回 SCHED_OTHER = 0"。

**文件路径**: `os/src/syscall/process/others.rs:161-165`

### 上下文切换实现（汇编分析）

上下文切换通过汇编函数 `__switch` 实现，保存 callee-saved 寄存器到 `TaskContext` 结构体。

#### LoongArch64 实现

**文件路径**: `os/src/arch/loongarch64/qemu/asms/switch.S:9-36`

```asm
.globl __switch
__switch:
    # save kernel stack of current task
    st.d $sp, $a0, 8
    # save ra & s0~s8 of current execution
    st.d $ra, $a0, 0
    .set n, 0
    .rept 9
        SAVE_SN %n          # 保存 s0-s8 到 [a0 + (n+2)*8]
        .set n, n + 1
    .endr
    st.d $fp, $a0, (9+2)*8
    # restore ra & s0~s8 of next execution
    ld.d $ra, $a1, 0
    .set n, 0
    .rept 9
        LOAD_SN %n          # 从 [a1 + (n+2)*8] 恢复 s0-s8
        .set n, n + 1
    .endr
    ld.d $fp, $a1, 0x58
    # restore kernel stack of next task
    ld.d $sp, $a1, 8
    li.d $a0, 0             # 返回 0 表示正常切换
    jr $ra
```

**保存的寄存器**（共 11 个）：
- `ra`（返回地址）
- `sp`（内核栈指针）
- `s0-s8`（9 个 callee-saved 寄存器）
- `fp`（帧指针）

#### RISC-V 64 实现

**文件路径**: `os/src/arch/riscv64/qemu/asms/switch.S:9-30`

```asm
.globl __switch
__switch:
    sd sp, 8(a0)            # 保存 sp
    sd ra, 0(a0)            # 保存 ra
    .set n, 0
    .rept 12
        SAVE_SN %n          # 保存 s0-s11
        .set n, n + 1
    .endr
    ld ra, 0(a1)            # 恢复 ra
    .set n, 0
    .rept 12
        LOAD_SN %n          # 恢复 s0-s11
        .set n, n + 1
    .endr
    ld sp, 8(a1)            # 恢复 sp
    li a0, 0
    ret
```

**保存的寄存器**（共 14 个）：
- `ra`（返回地址）
- `sp`（内核栈指针）
- `s0-s11`（12 个 callee-saved 寄存器）

**关键特点**：
- **不保存 caller-saved 寄存器**（如 `a0-a7`、`t0-t6`），由编译器保证在调用 `switch()` 前保存。
- **不保存浮点/向量寄存器**，内核**不支持**浮点上下文切换。
- 返回值 `a0=0` 表示正常切换；若 `a0>0` 则表示被切换出的任务需要释放内核栈（用于 `__abandon`）。

```rust
pub fn sys_prlimit(pid: usize, resource: u32, new_limit: *const RLimit, old_limit: *mut RLimit) -> SyscallRet {
    const RLIMIT_NOFILE: u32 = 7;
    if resource != RLIMIT_NOFILE {
        return Ok(0);  // 其他资源类型直接返回成功
    }
    if pid == 0 {  // 仅支持当前进程
        let task = current_task().unwrap();
        let mut inner = task.inner_lock();

        if !old_limit.is_null() {

                rlim_cur: fd_table.get_soft_limit(),
                rlim_max: fd_table.get_hard_limit(),
            };
            memory_set.put_data(old_limit, lim);
        }
        if !new_limit.is_null() {
            let limit: RLimit = memory_set.get_data(new_limit);
            fd_table.set_limit(limit.rlim_cur, limit.rlim_max);
        }
    } else {
        unimplemented!("pid must equal zero");
    }
    Ok(0)
}
```

- **无控制终端**：**未实现**终端会话管理。

#### 线程与进程的区别

本操作系统中，**线程与进程共享相同的 TCB 结构**，区别仅在于：

| 特性 | 进程 | 线程 |
|------|------|------|
| **地址空间** | 独立 `MemorySet` | 共享父进程 `MemorySet`（CLONE_VM） |
| **文件表** | 可独立（!CLONE_FILES） | 共享（CLONE_FILES） |
| **信号表** | 独立 `SigTable` | 共享（CLONE_SIGHAND） |
| **PID** | 独立 PID | 与进程组 PID 相同（CLONE_THREAD） |
| **TID** | 独立 TID（=PID） | 独立 TID（≠PID） |

**关键代码**（`os/src/syscall/process/clone.rs:263-310`）：
```rust
if flags.contains(CloneFlags::CLONE_THREAD) {
    timer = Arc::clone(&parent_inner.timer);
    sig_mask = SigSet::empty();
    process = self.process.clone();  // 共享 Process

    process = Process::new(memory_set.clone(), sig_table.clone(), pid, ppid);  // 新 Process

```

**结论**：代码中**未区分** PCB 和 TCB，而是通过 `Process` 包含多个 `TaskControlBlock` 实现"进程-线程"层次结构。

```rust
pub fn sys_write(fd: usize, buf: *const u8, len: usize) -> SyscallRet {
    let task = current_task().unwrap();
    let inner = task.inner_lock();
    
    // 1. 检查 fd 有效性
    if fd >= inner.fd_table.len() {
        return Err(SysErrNo::EINVAL);
    }
    
    // 2. 获取文件对象
    if let Some(file) = &inner.fd_table.try_get(fd) {
        let process = task.process.inner_lock();
        let memory_set = process.get_locked_memory_set();
        let file: Arc<dyn File> = file.any();
        
        // 3. 检查写权限
        if !file.writable() {
            return Err(SysErrNo::EACCES);
        }
        
        // 4. 创建用户态缓冲区包装（类型安全）
        let buffer = UserBuffer::new_safe(&memory_set, buf, len, true);
        
        // 5. 释放锁避免死锁
        drop(inner); drop(memory_set); drop(process);
        
        // 6. 执行实际写操作
        let ret = file.write(buffer)?;
        Ok(ret)
    } else {
        Err(SysErrNo::EBADF)
    }
}
```

```
trap_handler (识别 Syscall) → syscall() 分发 → 
sys_write() → UserBuffer::new_safe() → file.write() → 返回
```

```rust
pub fn lazy_page_fault(&mut self, vpn: VirtPageNum, is_write: bool) -> bool {
    // 检查是否已映射
    if !self.page_table.translate(vpn).is_none() {
        return false;
    }
    
    // mmap 区域处理
    if let Some(area) = self.areas.iter_mut()
        .filter(|area| area.area_type == MapAreaType::Mmap)
        .find(...) {
        if is_write {
            mmap_write_page_fault(...);
        } else {
            mmap_read_page_fault(...);
        }
        return true;
    }
    
    // brk/stack 区域处理
    if let Some(area) = self.areas.iter_mut()
        .filter(|area| area.area_type == MapAreaType::Brk || area.area_type == MapAreaType::Stack)
        .find(...) {
        lazy_page_fault(vpn.into(), &mut self.page_table, area);
        return true;
    }
    false
}
```

```rust
            let process = task.process.inner_lock();
            let memory_set = process.get_locked_memory_set();
            let is_write = cause == Trap::Exception(Exception::StorePageFault);
            
            // 1. Lazy Allocation
            ok = memory_set.lazy_page_fault(VirtAddr::from(stval).floor(), is_write);
            
            // 2. COW (仅写操作)
            if !ok && is_write {
                ok = memory_set.cow_page_fault(VirtAddr::from(stval).floor());
            }
        }
        
        // 3. 失败则终止
        if !ok {
            warn!("[kernel] hart {} {:?} in application, bad addr = {:#x}", 
                  hartid, cause, stval);
            exit_current_and_run_next(-2);
        }
    }
    // ...
}
```

**本章总结**：
TatlinOS 实现了完整的中断、异常与系统调用处理机制，支持 RISC-V 64 和 LoongArch 64 双架构。关键特性包括：
1. **Trap 处理**：双架构独立的汇编入口，统一的 Rust 处理逻辑
2. **上下文保存**：548 字节（RISC-V）/ 533 字节（LoongArch）的 `TrapContext` 结构体
3. **系统调用**：134 个 syscall 定义，核心功能（文件 I/O、进程、内存、信号）已完整实现
4. **信号机制**：支持线程级/进程级/线程组级三种粒度，含用户自定义处理函数和 sigreturn 跳板
5. **缺页异常**：支持 Lazy Allocation 和 COW，但 SIGSEGV 信号未实现（直接终止进程）
6. **时钟中断**：每秒 1 次中断，支持 futex 定时器和任务调度

```rust
// os/src/syscall/memory.rs:21-82
pub fn sys_mmap(
    addr: usize, len: usize, prot: u32, flags: u32, fd: usize, off: usize,
) -> SyscallRet {
    let map_perm: MapPermission = MmapProt::from_bits(prot).unwrap().into();
    let flags = MmapFlags::from_bits(flags).expect("...");
    
    if fd == usize::MAX {
        if !flags.contains(MmapFlags::MAP_ANONYMOUS) {
            return Err(SysErrNo::EBADF);
        }
        // 匿名映射
        return Ok(memory_set.mmap(addr, len, map_perm, flags, None, usize::MAX));
    }
    
    if flags.contains(MmapFlags::MAP_ANONYMOUS) {
        // 标志位冲突处理
        let rv = memory_set.mmap(0, 1, MapPermission::empty(), flags, None, usize::MAX);
        insert_bad_address(rv);
        return Ok(rv);
    }
    
    // 文件映射：检查 fd 和权限
    let file = task_inner.fd_table.get(fd).file()?;
    if map_perm.contains(MapPermission::R) && !file.readable()
        || flags.contains(MmapFlags::MAP_SHARED)
            && map_perm.contains(MapPermission::W)
            && !file.writable()
    {
        return Err(SysErrNo::EPERM);
    }
    
    Ok(memory_set.mmap(addr, len, map_perm, flags, Some(file), off))
}
```

```rust
pub fn open(abs_path: &str, flags: OpenFlags, mode: u32) -> Result<FileClass, SysErrNo> {
    // 1. 检查设备文件
    if find_device(abs_path) {
        return Ok(FileClass::Abs(open_device_file(abs_path)?));
    }
    
    // 2. 检查特殊文件（/proc/interrupts）
    if abs_path == "/proc/interrupts" {
        return Ok(FileClass::Abs(StringFile::new(proc.export_intr_counter())));
    }
    
    // 3. 查找 Inode（先查缓存 FsIndex）
    let mut inode: Option<Arc<dyn Inode>> = None;
    if FsIndex::has_inode(abs_path) {
        inode = FsIndex::find_inode_idx(abs_path);
    } else {
        let found_res = superblock_root_inode().find(abs_path, flags, 0);
        if let Ok(t) = found_res {
            FsIndex::insert_inode_idx(abs_path, t.clone());
            inode = Some(t);
        }
    }
    
    // 4. 如果节点存在，创建 OSFile
    if let Some(inode) = inode {
        let (readable, writable) = flags.read_write();
        let osfile = OSFile::new(readable, writable, inode);
        if flags.contains(OpenFlags::O_APPEND) {
            osfile.lseek(0, SEEK_END)?;
        }
        if flags.contains(OpenFlags::O_TRUNC) {
            osfile.inode.truncate(0)?;
        }
        return Ok(FileClass::File(Arc::new(osfile)));
    }
    
    // 5. 如果节点不存在且 O_CREATE，创建文件
    if flags.contains(OpenFlags::O_CREATE) {
        return create_file(abs_path, flags, mode);
    }
    
    Err(SysErrNo::ENOENT)
}
```

3. **Ext4Inode.find**（`os/src/fs/ext4_lw/inode.rs:180-224`）：
```rust
fn find(&self, path: &str, flags: OpenFlags, loop_times: usize) -> Result<Arc<dyn Inode>, SysErrNo> {
    // 检查目录、常规文件、符号链接
    if file.check_inode_exist(path, InodeTypes::EXT4_DE_DIR) {
        Ok(Arc::new(Ext4Inode::new(path, InodeTypes::EXT4_DE_DIR)))
    } else if file.check_inode_exist(path, InodeTypes::EXT4_DE_REG_FILE) {
        if flags.contains(OpenFlags::O_DIRECTORY) {
            return Err(SysErrNo::ENOTDIR);
        }
        Ok(Arc::new(Ext4Inode::new(path, InodeTypes::EXT4_DE_REG_FILE)))
    } else if file.check_inode_exist(path, InodeTypes::EXT4_DE_SYMLINK) {
        // 符号链接解析（最大 5 次循环）
        if loop_times >= MAX_LOOPTIMES {
            return Err(SysErrNo::ELOOP);
        }
        let mut file_name = [0u8; 256];
        file.file_readlink(&mut file_name, 256)?;
        let abs_path = core::str::from_utf8(&file_name[..end]).unwrap();
        return self.find(&abs_path, flags, loop_times + 1);
    } else {
        Err(SysErrNo::ENOENT)
    }
}
```

**四大核心数据结构协同**：
1. **SuperBlock**（`Ext4SuperBlock`）：管理文件系统元数据，提供根 Inode
2. **Inode**（`Ext4Inode`）：表示文件元数据和操作接口
3. **Dentry**：**未显式实现**，路径解析由 `Inode.find` 直接处理
4. **File**（`OSFile`/`FileClass`）：表示打开的文件实例，包含 offset 状态

**poll/select 支持**：
- `sys_ppoll` 已实现（`os/src/syscall/fs.rs:1393-1466`），支持超时和信号掩码
- 各 File 实现均实现 `poll()` 方法，检查可读/可写状态
- **未实现** `sys_select`（搜索无结果）

**高级特性总结**：

## 第 7 章：设备驱动与硬件抽象

### 驱动框架与设备发现

本项目实现了**基于 Feature 条件编译的静态驱动框架**，而非动态设备发现机制。驱动框架位于 `os/src/drivers/` 目录下，采用 Trait 抽象 + 条件编译选择具体实现的设计模式。

#### 驱动 Trait 定义

驱动框架的核心 Trait 定义在 `os/src/drivers/device.rs`：

```rust
pub trait BaseDriver: Send + Sync {
    fn device_type(&self) -> DeviceType;
}

pub trait BlockDriver: BaseDriver {
    fn num_blocks(&self) -> usize;
    fn block_size(&self) -> usize;
    fn read_block(&mut self, block_id: usize, buf: &mut [u8]) -> DevResult;
    fn write_block(&mut self, block_id: usize, buf: &[u8]) -> DevResult;
    fn flush(&mut self) -> DevResult;
}
```

设备类型枚举定义了四种设备：
- `Block`：块存储设备（磁盘）
- `Char`：字符设备（串口）
- `Net`：网络设备
- `Display`：图形显示设备

#### 设备发现机制：**无动态发现，采用静态配置**

经代码验证，本项目**未实现 Device Tree 解析或 PCI 动态扫描**。设备地址通过硬编码的内存映射常量定义：

**LoongArch64 平台**（`os/src/arch/loongarch64/qemu/memory_layout.rs`）：
```rust

#[cfg(feature = "board")]
pub const UART_ADDR: usize = 0x8000_0000_1FE2_0000;  // 上板

pub const MMIO: &[(usize, usize)] = &[
    (UART_ADDR - 0x8000_0000_0000_0000, 0x1000),  // UART0
    (0x100e_0000, 0x1000),                         // power off
    (0x2000_0000, 0x10000000),                     // PCI
];
```

**RISC-V 64 平台**（`os/src/arch/riscv64/qemu/memory_layout.rs`）：
```rust
pub const MMIO: &[(usize, usize)] = &[
    (0x0010_0000, 0x00_1000),  // VIRT_TEST
    (0x0010_1000, 0x00_1000),  // VIRT_RTC
    (0x1000_1000, 0x00_1000),  // Virtio Block
];
```

**PCI 配置空间访问**：虽然 `os/src/drivers/virtio/pci.rs` 实现了 `pci_config_read`/`pci_config_write` 函数，但这仅用于**硬编码设备 (0, DEVICE, 0)** 的配置，并非完整的 PCI 总线扫描。代码中存在遍历总线/设备/功能的循环，但实际使用的设备是硬编码的 `DEVICE = 1`。

#### 驱动注册与初始化

**未发现统一的驱动注册机制**。驱动初始化通过 `os/src/drivers/mod.rs` 中的条件编译在编译期确定：


#### MMU 启用前的串口访问

**RISC-V 64 平台**：通过 SBI 调用访问串口（`os/src/arch/riscv64/qemu/console.rs`）：

**LoongArch64 平台**：直接访问 MMIO 寄存器（`os/src/arch/loongarch64/qemu/console.rs`）：

```rust
struct Uart16550 {
    base_address: usize,
}

impl Uart16550 {
    pub fn putchar(&mut self, c: u8) {
        let ptr = self.base_address as *mut u8;
        loop {
            unsafe {
                let c = ptr.add(5).read_volatile();
                if c & (1 << 5) != 0 {  // 检查 THRE 位
                    break;
                }
            }
        }
        unsafe {
            ptr.add(0).write_volatile(c);
        }
    }
}
```

#### MMU 启用后的地址切换

**关键发现**：本项目**未显式处理 MMU 前后串口地址切换**。UART 基地址 `UART_ADDR` 在 `memory_layout.rs` 中定义为**直接映射的虚拟地址**：

```rust
// LoongArch64
pub const KERNEL_ADDR_OFFSET: usize = 0x9000_0000_0000_0000;
pub const UART_ADDR: usize = 0x8000_0000_1FE0_01E0;  // 物理地址
// 实际访问时使用：UART_ADDR + KERNEL_ADDR_OFFSET (通过直接映射窗口)
```

LoongArch64 架构通过设置 `CRMD.DA` 位启用直接映射窗口，物理地址 `0x0000_xxxx_xxxx_xxxx` 自动映射到虚拟地址 `0x9000_xxxx_xxxx_xxxx`。因此，**UART 驱动始终使用虚拟地址访问**，无需显式切换。

### 块设备驱动（VirtIO-Blk 等）

本项目实现了三种块设备驱动：

#### 1. 内存盘驱动（RamBlkDev）

位置：`os/src/drivers/ramdisk/mod.rs`

```rust
global_asm!(include_str!("ramdisk.S"));

extern "C" {
    fn ramdisk_start();
    fn ramdisk_end();
}

impl BlockDriver for RamBlkDev {
    fn read_block(&mut self, block_id: usize, buf: &mut [u8]) -> DevResult {
        let block = unsafe {
            core::slice::from_raw_parts(
                ((ramdisk_start as usize) + (512 * block_id)) as *const u8,
                512,
            )
        };
        buf.copy_from_slice(block);
        Ok(())
    }
    
    fn write_block(&mut self, block_id: usize, buf: &[u8]) -> DevResult {
        let block = unsafe {
            core::slice::from_raw_parts_mut(
                ((ramdisk_start as usize) + (512 * block_id)) as *mut u8,
                512,
            )
        };
        block.copy_from_slice(buf);
        Ok(())
    }
}
```

#### 支持的平台

| 架构 | 平台 | Feature 组合 | 块设备 | UART |
|------|------|-------------|--------|------|

| 常量 | RISC-V 64 | LoongArch 64 |

**启动流程差异**：

- **RISC-V 64**：通过 SBI 调用访问固件服务（串口、关机等）
- **LoongArch 64**：直接访问 MMIO 寄存器，需要手动设置直接映射窗口

```rust
// os/src/task/futex.rs:73-115
fn futex_wait_bitset(
    pa: usize,
    task: Arc<TaskControlBlock>,
    bitset: u32,
    timeout: Option<Timespec>,
) -> SyscallRet {
    let mut waitq = FUTEX_QUEUE_BITMAP.lock();
    let futex_key = new_futex_key();
    let waiter = FutexWaiter {
        task: Arc::downgrade(&task),
        bitset: bitset,
        futex_key: futex_key,
    };
    // 设置任务的 futex 状态
    let mut inner = task.inner_lock();
    inner.futex_pa = pa;
    inner.futex_key = futex_key;
    drop(inner);
    
    // 可选：添加超时定时器
    if let Some(timeout) = timeout {
        add_futex_timer(timeout, &task, futex_key);
    }
    
    // 插入等待队列
    if let Some(queue) = waitq.get_mut(&pa) {
        queue.push_back(waiter);
    } else {
        waitq.insert(pa, {
            let mut queue = VecDeque::new();
            queue.push_back(waiter);
            queue
        });
    }
    
    drop(waitq);
    // 阻塞当前任务并调度下一任务
    block_current_and_run_next();
    
    // 唤醒后检查是否被信号中断
    let task = current_task().unwrap();
    let task_inner = task.inner_lock();
    if !task_inner.sig_pending.difference(task_inner.sig_mask).is_empty() {
        return Err(SysErrNo::EINTR);
    }
    Ok(0)
}
```

```rust
fn futex_wake_up_bitset(pa: usize, max_num: i32, bitset: u32) -> usize {
    let mut futex_queue = FUTEX_QUEUE_BITMAP.lock();
    let mut num: usize = 0;
    if let Some(queue) = futex_queue.get_mut(&pa) {
        let queue_len = queue.len();
        let mut cnt: usize = 0;
        while cnt < queue_len && num < (max_num as usize) {
            cnt += 1;
            if let Some(waiter) = queue.pop_front() {
                if let Some(task) = waiter.task.upgrade() {
                    if bitset & waiter.bitset != 0 {
                        wakeup_futex_task(task);
                        num += 1;
                    } else {
                        queue.push_back(waiter); // 不匹配则放回
                    }
                }
            }
        }
    }
    num
}
```

```rust
pub fn send_signal_to_thread_group(pid: usize, sig: SigSet) -> Result<usize, SysErrNo> {
    let process = Process::get_process_arc_by_pid(pid);
    if let Some(proc) = process {
        let tasks = &proc.meta_lock().tasks;
        for task in tasks.iter() {
            if let Some(task) = task.upgrade() {
                add_signal(&task, sig);
            }
        }
        return Ok(0);
    } else {
        return Err(SysErrNo::ESRCH);
    }
}
```

```rust
// os/src/task/futex.rs:174-283
pub fn sys_futex(
    uaddr: *mut i32,
    futex_op: u32,
    val: i32,
    timeout: *const Timespec,
    uaddr2: *mut u32,
    _val3: i32,
) -> SyscallRet {
    let cmd = FutexCmd::try_from(futex_op & 0x7f).expect("invalid futex op");
    let opt = FutexOpt::from_bits_truncate(futex_op);
    
    // 检查 uaddr 对齐
    if uaddr.align_offset(4) != 0 {
        return Err(SysErrNo::EINVAL);
    }
    
    let task = current_task().unwrap();
    let process = task.process.inner_lock();
    let memory_set = process.get_locked_memory_set();
    
    // 虚拟地址转物理地址
    let pa = memory_set
        .translate_va(VirtAddr::from(uaddr as usize))
        .unwrap()
        .0;
    
    // 处理超时参数
    let timeout_opt: Option<Timespec>;
    if timeout.is_null() || !(cmd == FutexCmd::Wait || cmd == FutexCmd::WaitBitset) {
        timeout_opt = None;
    } else {
        // 转换用户态超时为内核态绝对时间
        let mut real_timeout = memory_set.get_data(timeout);
        if opt.contains(FutexOpt::FUTEX_CLOCK_REALTIME) {
            real_timeout.tv_sec -= crate::timer::NOW_TIME_STAMP;
        } else {
            real_timeout = real_timeout + get_time_spec();
        }
        timeout_opt = Some(real_timeout);
    }
    
    let val_at_uaddr = memory_set.get_data(uaddr);
    drop(memory_set);
    drop(process);
    
    match cmd {
        FutexCmd::Wait | FutexCmd::WaitBitset => {
            // 检查用户态值是否等于期望值
            if val_at_uaddr != val {
                return Err(SysErrNo::EAGAIN);
            }
            let bitset = if cmd == FutexCmd::Wait { u32::MAX } else { _val3 as u32 };
            futex_wait_bitset(pa, task, bitset, timeout_opt)
        }
        FutexCmd::Wake | FutexCmd::WakeBitset => {
            drop(task);
            let bitset = if cmd == FutexCmd::Wake { u32::MAX } else { _val3 as u32 };
            Ok(futex_wake_up_bitset(pa, val, bitset))
        }
        FutexCmd::Requeue => {
            drop(task);
            if let Some(pa2) = pa2 {
                return Ok(futex_requeue(pa, val, pa2.0, timeout as i32));
            } else {
                return Err(SysErrNo::EINVAL);
            }
        }
        _ => unimplemented!(),
    }
}
```

#### 管道读写完整实现

```rust
// os/src/fs/files/pipe.rs:197-270
impl File for Pipe {
    fn read(&self, mut buf: UserBuffer) -> SyscallRet {
        assert!(self.readable());
        let mut read_size = 0usize;
        let mut loop_read;
        
        // 等待数据可用
        loop {
            let ring_buffer = self.inner_lock();
            loop_read = ring_buffer.available_read();
            if loop_read == 0 {
                if ring_buffer.all_write_ends_closed() {
                    return Ok(read_size); // EOF
                }
                drop(ring_buffer);
                suspend_current_and_run_next(); // 阻塞
                continue;
            } else {
                break;
            }
        }
        
        // 读取数据
        let mut ring_buffer = self.inner_lock();
        let length = buf.len();
        if length <= 10 {
            let mut buf_iter = buf.into_iter();
            for _ in 0..loop_read {
                if let Some(byte_ref) = buf_iter.next() {
                    unsafe { *byte_ref = ring_buffer.read_byte(); }
                    read_size += 1;
                } else {
                    break;
                }
            }
        } else {
            read_size = min(loop_read, length);
            buf.write(&ring_buffer.read_bytes(read_size));
        }
        Ok(read_size)
    }
    
    fn write(&self, mut buf: UserBuffer) -> SyscallRet {
        assert!(self.writable());
        let mut write_size = 0usize;
        let mut loop_write;
        
        // 等待缓冲区有空间
        loop {
            let ring_buffer = self.inner_lock();
            loop_write = ring_buffer.available_write();
            if loop_write == 0 {
                drop(ring_buffer);
                suspend_current_and_run_next(); // 阻塞
                continue;
            } else {
                break;
            }
        }
        
        // 写入数据
        let mut ring_buffer = self.inner_lock();
        let length = buf.len();
        if length <= 10 {
            let mut buf_iter = buf.into_iter();
            for _ in 0..loop_write {
                if let Some(byte_ref) = buf_iter.next() {
                    ring_buffer.write_byte(unsafe { *byte_ref });
                    write_size += 1;
                } else {
                    break;
                }
            }
        } else {
            write_size = min(loop_write, length);
            ring_buffer.write_bytes(&buf.read(write_size), write_size);
        }
        Ok(write_size)
    }
}
```

## 第 10 章：安全机制与权限模型

本章分析 TatlinOS 的安全隔离与权限控制机制，涵盖用户/内核态隔离、权限模型、内存安全检查、安全沙箱等关键安全特性。

### 特权级与隔离机制

#### 硬件特权级隔离

TatlinOS 支持 **RISC-V64** 和 **LoongArch64** 两种架构，均通过硬件页表机制实现用户态与内核态的隔离。

**RISC-V64 架构**：
- 页表项通过 `USER` 位（bit 4）区分用户/内核页面
- 用户页面设置 `RVPTEFlags::USER`，内核页面不设置
- 页表权限转换见 `os/src/arch/riscv64/qemu/page_table.rs:47-85`

```rust
// os/src/arch/riscv64/qemu/page_table.rs:47-65
impl From<MapPermission> for RVPTEFlags {
    fn from(perm: MapPermission) -> Self {
        let mut flags = RVPTEFlags::VA;
        if perm.contains(MapPermission::R) {
            flags.insert(RVPTEFlags::READABLE);
        }
        if perm.contains(MapPermission::W) {
            flags.insert(RVPTEFlags::WD);
        }
        if perm.contains(MapPermission::X) {
            flags.insert(RVPTEFlags::EXECUTABLE);
        }
        if perm.contains(MapPermission::U) {  // 用户态访问位
            flags.insert(RVPTEFlags::USER);
        }
        flags
    }
}
```

**LoongArch64 架构**：
- 使用 `PLV3`（Privilege Level 3）标识用户页面
- 通过 `UNREADEABLE`、`UNEXECUTABLE` 位控制访问权限
- 页表权限转换见 `os/src/arch/loongarch64/qemu/page_table.rs:76-115`

```rust
// os/src/arch/loongarch64/qemu/page_table.rs:76-95
impl From<MapPermission> for LAPTEFlags {
    fn from(perm: MapPermission) -> Self {
        let mut flags = LAPTEFlags::VALID | LAPTEFlags::MAT_CC | LAPTEFlags::P;
        if !perm.contains(MapPermission::R) {
            flags.insert(LAPTEFlags::UNREADEABLE);
        }
        if perm.contains(MapPermission::W) {
            flags.insert(LAPTEFlags::WRITEABLE);
        }
        if !perm.contains(MapPermission::X) {
            flags.insert(LAPTEFlags::UNEXECUTABLE);
        }
        if perm.contains(MapPermission::U) {
            flags.insert(LAPTEFlags::PLV3);  // 用户特权级
        }
        flags
    }
}
```

#### 地址空间隔离

用户进程拥有独立的地址空间，通过 `MemorySet` 管理：
- 每个进程有独立的页表（`satp` 寄存器切换）
- 内核空间在所有进程中映射（通过 `KERNEL_SPACE`）
- 用户空间通过 `MapPermission::U` 位限制内核访问

**证据**：`os/src/mm/memory_set/inner.rs` 中每个进程创建时分配独立 `MemorySet`

```rust
    if inner.user_id != 0
```

**安全问题**：

### 内存安全与系统调用检查

#### 用户指针验证机制

TatlinOS **未使用** `access_ok`、`verify_area` 等显式验证函数，而是通过 `UserBuffer` 机制在访问时进行页表翻译验证。

**`UserBuffer` 实现**：

```rust
// os/src/mm/user_buffer.rs:16-40
fn translated_byte_buffer(
    token: usize,
    ptr: *const u8,
    len: usize,
) -> Option<&'static mut [u8]> {
    let page_table = PageTable::from_token(token);
    let mut start = ptr as usize;
    let end = start + len;
    let mut v = Vec::new();
    while start < end {
        let start_va = VirtAddr::from(start);
        let mut vpn = start_va.floor();
        let ppn = match page_table.translate(vpn) {
            None => {
                debug!("vpn {:#x} not found", vpn.0);
                return None;  // 翻译失败返回 None
            }
            Some(ppn) => ppn,
        };
        // ... 构建缓冲区 ...
    }
    Some(v)
}
```

**安全特性**：
1. **按需翻译**：通过页表翻译验证用户指针有效性
2. **跨页处理**：正确处理跨页缓冲区
3. **失败处理**：翻译失败返回 `None`，调用方应检查

**`new_safe` 方法**：支持惰性分配（缺页时自动分配）

```rust
// os/src/mm/user_buffer.rs:113-118
pub fn new_safe(memory_set: &MemorySet, ptr: *const u8, len: usize, readonly: bool) -> Self {
    let buf = memory_set.safe_translated_byte_buffer(ptr, len, readonly);
    Self {
        buffers: buf,
        readonly: readonly,
    }
}
```

**局限性**：
- **未验证指针范围**：仅检查页表映射，未验证指针是否越界
- **无 `copy_from_user` 封装**：直接使用 `UserBuffer`，缺少统一的安全抽象

### Rust 语言级安全性机制

#### 所有权与生命周期

TatlinOS 使用 Rust 编写，利用以下语言特性增强安全性：

1. **RAII 资源管理**：
   - `Arc<T>` 用于共享资源（如 `Process`、`FdTable`）
   - `Mutex<T>`、`RwLock<T>` 用于同步访问
   - 资源在作用域结束时自动释放

2. **可变性控制**：
   - 默认不可变，需显式使用 `mut`
   - 防止意外修改

3. **类型安全**：
   - 强类型系统防止类型混淆
   - `bitflags!` 宏确保标志位类型安全

#### 锁机制

**自旋锁**：使用 `spin::Mutex` 和 `spin::RwLock`（无操作系统依赖）

### 安全机制总结

**已实现的安全特性**：
1. ✅ 硬件特权级隔离（RISC-V U 模式 / LoongArch PLV3）
2. ✅ 进程地址空间隔离（独立页表）
3. ✅ UID 字段存储与读取（`sys_getuid`/`sys_setuid`）
4. ✅ 基础文件权限检查（`sys_faccessat`，仅检查模式位）
5. ✅ 用户指针页表翻译验证（`UserBuffer` 机制）
6. ✅ Rust 语言级内存安全（所有权、类型安全）

**主要安全缺陷**：
1. ⚠️ **UID 切换无限制**：`sys_setuid` 允许任意提权
2. ⚠️ **权限检查不完整**：仅 `faccessat` 有检查，`open`/`read`/`write` 无检查
3. ⚠️ **硬编码返回值**：`sys_geteuid`/`sys_getgid` 始终返回 0
4. ⚠️ **无安全沙箱**：Seccomp/Prctl 未实现
5. ⚠️ **无审计机制**：安全事件无法追踪
6. ⚠️ **无栈保护**：易受栈溢出攻击

**适用场景评估**：
- 适用于教学/实验环境
- **不适用于生产环境**（缺少关键安全机制）
- 需补充完整的权限检查、安全沙箱、审计日志等功能

### `sys_socket` 实现分析

```rust
// os/src/syscall/net.rs:25-56
pub fn sys_socket(_domain: u32, _type: u32, _protocol: u32) -> SyscallRet {
    let domain: SocketDomain = SocketDomain::from(_domain);
    let task = current_task().unwrap();
    let task_inner = task.inner_lock();
    let new_fd = task_inner.fd_table.alloc_fd()?;
    // 创建 SimpleSocket（实际是 Pipe 包装）
    task_inner.fd_table.set(
        new_fd,
        FileDescriptor::new(flags, FileClass::Abs(make_socket())),
    );
    Ok(new_fd)
}
```

### `SimpleSocket` 本质是管道

```rust
// os/src/fs/files/socket.rs:11-16
pub fn make_socket() -> Arc<dyn File> {

}
```

### `sys_sendto` / `sys_recvfrom` 限制

```rust
// os/src/syscall/net.rs:149-168
pub fn sys_sendto(...) -> SyscallRet {
    // 强制检查目标地址必须为 127.0.0.1
```

## Panic 处理与栈回溯

### Panic Handler 实现

Panic 处理器定义在 `os/src/lang_items.rs`，当 Rust 代码触发 `panic!` 时会被调用：

```rust
// os/src/lang_items.rs:22-35
#[panic_handler]
fn panic(info: &PanicInfo) -> ! {
    println!("panic");
    if let Some(location) = info.location() {
        println!(
            "[kernel] Panicked at {}:{} {}",
            location.file(),
            location.line(),
            info.message().unwrap()
        );
    } else {
        println!("[kernel] Panicked: {}", info.message().unwrap());
    }
    shutdown(true)
}
```

**Panic 处理流程**：
1. 打印 "panic" 标识
2. 输出 panic 位置（文件、行号、消息）
3. 调用 `shutdown(true)` 停机

### 停机实现

停机函数因架构而异：

**RISC-V 架构** (`os/src/arch/riscv64/qemu/cpu.rs:18-25`)：
```rust
pub fn shutdown(failure: bool) -> ! {
    if !failure {
        system_reset(Shutdown, NoReason);
    } else {
        system_reset(Shutdown, SystemFailure);
    }
    unreachable!()
}
```

**LoongArch64 架构** (`os/src/arch/loongarch64/qemu/cpu.rs:25-30`)：
```rust
pub fn shutdown(_failure: bool) -> ! {
    if cfg!(not(feature = "board")) {
        unsafe {
            ((0x9000_0000_100e_001c) as *mut u8).write_volatile(0x34);
        }
    }
    loop {}
}
```

在 QEMU 环境下，LoongArch64 通过写入特定 MMIO 地址触发关机；在板级环境下进入死循环。

### 栈回溯 (Backtrace) 支持情况

**关键发现**：TatlinOS 的栈回溯功能**仅有框架，未实现实际功能**。

在 `os/src/utils/mod.rs` 中定义了 `backtrace()` 函数：


## 本章总结

| 功能 | 支持状态 | 说明 |
|------|----------|------|
| 日志系统 | ✅ 已实现 | 支持 5 级别，编译时配置，带 PID/TID 上下文 |
| Panic 处理 | ✅ 已实现 | 打印位置信息后停机 |
| 栈回溯 (Backtrace) | ❌ 未实现 | 函数存在但代码被注释 |
| DWARF 解析 | ❌ 未实现 | 未找到相关代码 |
| 交互式 Shell | ❌ 未实现 | 仅用户空间 busybox |
| GDB Stub | ❌ 未实现 | 无数据包解析代码 |
| Perf/Ftrace | ❌ 未实现 | 无 tracepoints 支持 |
| 错误码设计 | ✅ 已实现 | 完整 Linux 兼容错误码 |
| 断言检查 | ✅ 已实现 | 广泛使用 assert! |

**核心结论**：
1. TatlinOS 具备基础的日志和 panic 处理能力，但**缺乏高级调试功能**
2. **栈回溯功能未实现**，panic 时仅打印 panic 位置，不打印调用栈
3. **无内核级调试接口**（无 Monitor、无 GDB Stub、无 Shell）
4. 错误处理机制完善，采用标准 Linux 错误码体系
5. 调试主要依赖外部工具（QEMU GDB 服务器）和日志输出

```rust
pub fn run_ltp_tests_musl(tests: &[&str], blacklist: &[&str]) {
    println!("#### OS COMP TEST GROUP START ltp-musl ####");

        let trim_test = test.trim_end_matches('\0');
```

### 设计亮点

#### 1. 双架构统一硬件抽象层（HAL）

项目通过条件编译与 Trait 抽象实现了 RISC-V/LoongArch 双架构的统一接口：

```rust

    pub fn map(&mut self, vpn: VirtPageNum, ppn: PhysPageNum, flags: PTEFlags) { }
    pub fn translate(&self, vpn: VirtPageNum) -> Option<PhysPageNum> { }
    pub fn handle_cow_page_fault(&mut self, va: VirtAddr, vma: &mut MapArea) -> bool { }

```

**优势**：上层内存管理代码（`os/src/mm/`）无需关心架构差异，仅需调用统一接口。RISC-V 使用 SV39 三级页表，LoongArch 使用 TLB+ 硬件页表遍历，但对外暴露相同 API。

#### 2. RAII 资源管理与物理帧追踪

通过 `FrameTracker` 结构体实现物理帧的生命周期管理：

```rust

pub struct FrameTracker {
    pub ppn: PhysPageNum,
}

impl Drop for FrameTracker {
    fn drop(&mut self) {
        PAGE_CACHE.dealloc(self.ppn);  // 析构时自动回收
    }
}
```

**优势**：利用 Rust 所有权系统，物理帧在 `Arc<FrameTracker>` 引用计数归零时自动回收，避免内存泄漏。CoW 场景中通过 `Arc::strong_count()` 判断是否需要复制页面。

### 不足与改进空间

#### 1. 多核支持缺失（架构级限制）

**问题**：`os/src/config.rs` 硬编码 `HART_NUM = 1`，`boot_all_harts()` 函数被完全注释，Secondary CPU 无法启动。

**影响**：无法利用多核并行性，性能受限；`UPSafeCell` 等单核原语在多核下会产生数据竞争。

**改进建议**：
- 实现 SBI `hart_start` 调用启动从核
- 将 `UPSafeCell` 替换为中断安全的 `SpinLock`
- 实现 IPI 机制用于 TLB Shootdown 和核间通信
- 添加多核调度策略（负载均衡、CPU 亲和性）

#### 2. 网络功能仅为桩实现

**问题**：`os/src/fs/files/socket.rs` 中 `SimpleSocket` 基于两个 Pipe 模拟，`sys_sendto` 仅将数据推入全局 `UDP_QUEUE`，无真实协议栈。

**影响**：无法进行真实网络通信，仅支持本地回环（127.0.0.1）；网络相关系统调用（bind/connect/listen）均返回 `Ok(0)` 无实际逻辑。

**改进建议**：
- 集成 `smoltcp` 协议栈（文档提及但未合并）
- 实现 VirtIO-Net 网卡驱动
- 完善 Socket 状态机（LISTEN/ESTABLISHED/CLOSED）

#### 3. 安全机制不完善

**问题**：
- `sys_setuid` 允许任意进程切换到任意 UID，无权限检查
- `sys_geteuid`/`sys_getgid` 硬编码返回 0，与实际 `user_id` 不一致
- 仅 `sys_faccessat` 有基础权限检查，`open`/`read`/`write` 无检查

**影响**：任意进程可提权至 root，文件系统无访问控制，不适合多用户环境。

**改进建议**：
- 实现完整的 UID/GID 权限检查（比对文件 `st_uid` 与进程 `user_id`）
- 限制 `sys_setuid` 仅允许 root 切换 UID 或进程切换至自身 UID
- 实现 Capability 机制细化权限控制

#### 4. 调试功能薄弱

**问题**：
- `backtrace()` 函数实现代码被完全注释
- 无 GDB Stub 支持（仅依赖 QEMU 外部 GDB 服务器）
- 无内核交互式 Shell/Monitor

**影响**：内核崩溃时无法获取完整调用栈，调试效率低。

**改进建议**：
- 实现基于 FramePointer 的栈回溯
- 集成 `gdbstub` crate 提供内核级 GDB 支持
- 添加简易 Kernel Monitor 支持内存查看/寄存器转储

#### 5. 测试自动化不足

**问题**：
- 主项目无 CI/CD 配置（仅 `lwext4_rust` 子模块有 GitHub Actions）
- 单元测试仅 1 个（`lwext4_rust/src/lib.rs` 中的 `it_works`）
- 测试结果依赖人工分析串口输出，未自动化收集

**改进建议**：
- 为主项目添加 GitHub Actions/GitLab CI 配置
- 增加内核模块单元测试（使用 `#[test]`）
- 集成 QEMU 自动化测试脚本，收集测试结果并生成报告
