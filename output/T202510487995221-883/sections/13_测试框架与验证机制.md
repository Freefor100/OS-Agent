### 单元测试与集成测试框架

#### Rust 单元测试

本项目中 Rust 代码的单元测试覆盖较少。通过 `grep_in_repo` 搜索 `#[test]` 和 `#[cfg(test)]`，仅发现 **1 个测试函数**，位于 `lwext4_rust/src/lib.rs`：

```rust
// lwext4_rust/src/lib.rs:28-37
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn it_works() {
        let result = add(2, 2);

    }

```

这是一个基础的示例测试，验证 `add()` 函数的正确性。除 `lwext4_rust` 子模块外，主 OS 内核 (`os/`) 和用户态库 (`user/`) 中**未发现**使用 `#[test]` 标记的单元测试。

#### 用户态集成测试

项目实现了较为完整的用户态集成测试框架，测试代码位于 `user/src/bin/` 目录下，按测试阶段分为：

- **`basic/`**：基础系统调用测试，包含 32 个测试用例（`brk`、`fork`、`mmap`、`pipe` 等）
- **`pre_tests/`**：预测试套件，执行快速验证
- **`final_tests/`**：最终测试套件，执行完整功能验证
- **`ltp/`**：Linux Test Project (LTP) 移植测试
- **`lmbench/`**：性能基准测试
- **`libctest/`**：musl libc 兼容性测试
- **`git/`**：Git 工具链测试

测试执行框架通过 `run_testsuit()` 函数调用 shell 脚本，支持 musl 和 glibc 双运行时环境。例如 `user/src/bin/pre_tests/mod.rs` 中：

```rust
// user/src/bin/pre_tests/mod.rs:9-25
#[no_mangle]
#[cfg(feature = "loongarch64")]
pub fn main_pre() {
    // basic
    run_testsuit("musl\0", "basic_testcode.sh\0");
    run_testsuit("glibc\0", "basic_testcode.sh\0");

    // busybox
    run_testsuit("musl\0", "busybox_testcode.sh\0");
    run_testsuit("glibc\0", "busybox_testcode.sh\0");

    // libctest (不含 glibc)
    run_testsuit("musl\0", "libctest_testcode.sh\0");

    // ltp
    ltp::run_ltp_tests_musl(ltp::MUSL_WRITE_LIST, ltp::LOONGARCH_MUSL_PANIC_LIST);
    ltp::run_ltp_tests_glibc(ltp::GLIBC_WRITE_LIST, ltp::LOONGARCH_GLIBC_PANIC_LIST);
}
```

#### LTP 测试支持

项目移植了 LTP (Linux Test Project) 测试套件，测试用例列表定义在 `user/src/bin/ltp/filelist.rs` 中（2823 行，64.6KB）。为避免已知问题导致测试卡死，项目实现了黑名单机制：

- **`blacklist.rs`** (1852 行，146.6KB)：按架构和运行时环境分类的黑名单
  - `LOONGARCH_MUSL_PANIC_LIST`：LoongArch + musl 环境下会卡死的测试
  - `LOONGARCH_GLIBC_PANIC_LIST`：LoongArch + glibc 环境下会卡死的测试
  - `RISCV_MUSL_PANIC_LIST`：RISC-V + musl 环境下会卡死的测试
  - `RISCV_GLIBC_PANIC_LIST`：RISC-V + glibc 环境下会卡死的测试

黑名单中包含的测试用例及原因示例：

```rust
// user/src/bin/ltp/blacklist.rs:50-60
"fork14",  // 卡死 (错误：FAIL LTP CASE fork13: 2...)
"fork_procs", // 卡死 (错误：FAIL LTP CASE fork_freeze.sh: 2...)
"fs_racer_dir_test.sh", // 无限循环，无法停止
"ftest04",   // 卡死 (错误：FAIL LTP CASE ftest03 : 1...)
"ftest08",   // 卡死 (错误：FAIL LTP CASE ftest07 : 1...)
```

LTP 测试执行逻辑位于 `user/src/bin/ltp/mod.rs`：

```rust
// user/src/bin/ltp/mod.rs:11-23
pub fn run_ltp_tests_musl(tests: &[&str], blacklist: &[&str]) {
    println!("#### OS COMP TEST GROUP START ltp-musl ####");
    for &test in tests {
        let trim_test = test.trim_end_matches('\0');
        if blacklist.contains(&trim_test) {
            continue;
        }
        println!("RUN LTP CASE {}", trim_test);
        let r = fork_and_run("/musl/ltp/testcases/bin\0", &[test]);
        println!("FAIL LTP CASE {} : {}", trim_test, r);
    }
    println!("#### OS COMP TEST GROUP END ltp-musl ####");
}
```

**注意**：输出中的 "FAIL" 并非表示测试失败，而是打印测试用例的返回值。

##### IOZone 文件系统测试

项目支持 IOZone 文件系统性能测试，通过 `iozone_testcode.sh` 脚本执行。根据 `Docs/TrustOsDocs/iozone.md` 记录，曾遇到 `lseek` 参数处理问题：

> **问题**：lseek 返回 INVALID ARGUMENT  
> **原因**：lseek 未正确处理从末尾往回计算 offset 的情况  
> **解决方法**：去掉对 offset 小于 0 的判断，对 SEEK_CUR 和 SEEK_END 两种情况添加小于 0 的 offset 计算方法

#### 已知测试问题

根据 `Docs/TrustOsDocs/` 目录下的文档，记录了以下测试相关问题及解决方案：

**LTP 测试问题** (`Docs/TrustOsDocs/ltp.md`)：

- `bw_pipe` 卡死：管道缓冲区需扩展至 65536 字节
- 信号处理：需注册 SIGQUIT 等信号的内核处理函数

**libc 测试问题** (`Docs/TrustOsDocs/libc_test.md`)：
- `pipe error`：fscanf 阻塞读取，需将 pipe 改为非阻塞实现
- `sys_unlinkat error`：ext4 文件链接计数与 fd 管理问题
- `sys_munmap error`：mmap 区间拆分回收问题
- `sys_mmap` 重复映射卡死：MAP_FIXED 标志位处理
- 信号处理 LoadPageFault：需实现 SA_SIGINFO 标志位支持

#### 测试验证机制

项目采用**运行时验证**而非编译时测试：
1. 测试程序在 QEMU 中运行
2. 测试结果输出到串口控制台
3. 通过人工或脚本分析控制台输出
4. 黑名单机制跳过已知问题测试

**局限性**：
- 无自动化测试结果收集
- 无测试覆盖率统计工具（如 cargo-tarpaulin、gcov）
- 无持续集成环境中的自动化测试执行（主项目）
- 测试结果未持久化保存

### 总结

本项目的测试框架具有以下特点：

**优势**：
1. 移植了 LTP、LMBench 等成熟测试套件，覆盖系统调用、文件系统、性能基准
2. 实现了 musl/glibc 双运行时环境的测试支持
3. 建立了完善的黑名单机制，避免已知问题导致测试卡死
4. lwext4_rust 子模块具有完整的 GitHub Actions CI 配置

**不足**：
1. **主项目无 CI/CD 配置**，依赖人工运行测试
2. **单元测试覆盖率低**，仅 lwext4_rust 有 1 个测试函数
3. **无模糊测试**和内存安全检测工具集成
4. **测试结果未自动化收集**，依赖人工分析串口输出
5. 部分性能测试（cyclictest、netperf、iperf）**尚未实现**

**改进建议**：
1. 为主项目添加 GitHub Actions/GitLab CI 配置
2. 增加内核模块的单元测试（使用 `#[test]`）
3. 集成 QEMU 自动化测试脚本，收集测试结果
4. 引入 cargo-tarpaulin 等工具统计代码覆盖率
5. 实现未完成的性能测试套件
