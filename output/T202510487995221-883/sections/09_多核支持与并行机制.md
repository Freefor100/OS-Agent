## 第 9 章：多核支持与并行机制

### 多核架构设计（SMP/AMP）

**结论：本项目为单核（UP, Uniprocessor）架构，不支持多核/SMP。**

关键证据：

1. **CPU 数量硬编码为 1**：
   ```rust
   // os/src/config.rs
   pub const HART_NUM: usize = 1;
   ```

2. **Processor 数组大小基于 `HART_NUM`**：
   ```rust
   // os/src/task/processor.rs
   pub static mut PROCESSORS: [Processor; HART_NUM] = [EMPTY_PROCESSOR; HART_NUM];
   ```

3. **hart_id() 函数硬编码返回 0**：
   ```rust
   // os/src/arch/loongarch64/qemu/cpu.rs
   pub fn hart_id() -> usize {
       // 我们是单核系统！
       0
   }
   ```

架构设计分析：
- 项目采用 **UP（单处理器）** 设计，而非 SMP（对称多处理）或 AMP（非对称多处理）
- 每个 CPU 核理论上应有独立的 `Processor` 结构体管理其调度状态，但由于 `HART_NUM=1`，实际上只有一个全局调度器
- 同步原语使用 `UPSafeCell`（基于 `RefCell`），**仅适用于单核环境**，在多核下会产生数据竞争

### Secondary CPU 启动流程

**结论：Secondary CPU 启动功能未实现，仅有被注释掉的占位代码。**

搜索结果显示：
- 未找到 `smp_boot`、`__cpu_up`、`start_secondary`、`smp_init` 等标准 SMP 启动符号
- `os/src/main.rs` 中存在被注释掉的 `boot_all_harts` 函数：

```rust
// os/src/main.rs:107-112
// pub fn boot_all_harts(hartid: usize) {
//     for i in (0..arch::config::HART_NUM).filter(|id| *id != hartid) {
//         let sbi_ret = arch::cpu::hart_start(i, arch::memory_layout::HART_START_ADDR).into_result();
//         match sbi_ret {
//             Ok(_) => (),
//             Err(_) => println!("Error when booting No.{} hart", i),
//         }
//     }
// }
```

该函数被完全注释，且由于 `HART_NUM=1`，即使取消注释也不会启动任何 secondary hart。

**启动流程分析（单核）**：
1. 所有 hart 从 `entry.asm` 进入，调用 `trampoline(hartid)`
2. `trampoline` 调整栈指针后跳转到 `rust_main(hartid)`
3. 第一个 hart（`FIRST_HART`）执行完整初始化：
   - `clear_bss()`、`trap::init()`、`mm::init()`、`task::init()`、`fs::init()`
   - 设置 `START_HART_ID` 和 `INIT_FINISHED`
4. 其他 hart（理论上）在 barrier 处等待 `INIT_FINISHED`，然后执行有限初始化：
   ```rust
   // os/src/main.rs:184-191
   } else {
       // barrier
       while !INIT_FINISHED.load(Ordering::SeqCst) {}
       println!("[kernel] ---------- hart {} is starting... ----------", hartid);
       trap::init();
       mm::activate_kernel_space();
       arch::trap_interface::enable_timer_interrupt();
       timer::set_next_trigger();
   }
   ```
   但由于 `HART_NUM=1`，此分支永远不会执行。

### 核间通信与 IPI 机制

**结论：IPI（核间中断）机制未实现。**

搜索结果：
- 仅在 `lwext4_rust/examples/src/boot/sbi.rs` 中找到 IPI 相关的 SBI 调用封装：
  ```rust
  pub fn clear_ipi(){
      sbi_call(SBI_CLEAR_IPI, 0, 0, 0);
  }
  pub fn send_ipi(sipi_value: usize){
      sbi_call(SBI_SEND_IPI, sipi_value, 0, 0);
  }
  const SBI_CLEAR_IPI: usize = 3;
  const SBI_SEND_IPI: usize = 4;
  ```
- **OS 主内核（`os/src/`）中未使用任何 IPI 相关函数**
- 未找到 `ipi_handler`、`send_ipi`、`broadcast_ipi` 等内核级 IPI 处理逻辑

由于是单核设计，IPI 机制无实现必要。

### Per-CPU 变量与数据结构

**结论：存在 Per-CPU 设计雏形，但仅支持单核。**

Per-CPU 数据结构：
```rust
// os/src/task/processor.rs
pub struct Processor {
    pub current: Option<Arc<TaskControlBlock>>,
    pub idle_task_cx: Option<Box<TaskContext>>,
}

pub static mut PROCESSORS: [Processor; HART_NUM] = [EMPTY_PROCESSOR; HART_NUM];
```

访问方式：
```rust
fn get_proc_by_hartid(hartid: usize) -> &'static mut Processor {
    if hartid >= HART_NUM {
        panic!("get_proc_by_hartid: fail because hartid={} is too large!", hartid);
    }
    unsafe { &mut PROCESSORS[hartid] }
}
```

**设计特点**：
1. **静态数组**：`PROCESSORS` 是固定大小的静态数组，每个元素对应一个 hart
2. **无锁访问**：注释说明"不需要加锁，每个核只会访问固定的 Processor"，这在单核下成立，但在多核下需要 Per-CPU 内存屏障
3. **未使用 `axns` 或类似 Per-CPU 命名空间**：搜索 `PerCpu`、`per_cpu`、`axns` 仅在文档中找到一处提及（`Percpu: 496 kB`），无实际实现

**原子操作使用**：
项目使用 `AtomicUsize` 进行跨线程安全的计数器操作：
```rust
// os/src/task/futex.rs
static FUTEX_KEY_COUNTER: AtomicUsize = AtomicUsize::new(0);
fn new_futex_key() -> usize {
    FUTEX_KEY_COUNTER.fetch_add(1, Ordering::Relaxed)
}

// os/src/main.rs
static FIRST_HART: AtomicBool = AtomicBool::new(true);
static INIT_FINISHED: AtomicBool = AtomicBool::new(true);
static START_HART_ID: AtomicUsize = AtomicUsize::new(0);
```

这些原子操作在单核下主要防止中断上下文竞争，在多核下可提供基本的内存序保证（但项目未使用 `Acquire`/`Release` 语义）。

### 多核调度策略

**结论：无多核调度策略，采用单核抢占式调度。**

调度器设计：
- 每个 hart 有独立的 `Processor` 结构管理当前任务
- 全局就绪队列 `ready_queue`（未在此章详细分析，见第 4 章）
- 调度函数 `schedule()` 仅操作当前 hart 的 `Processor`

```rust
// os/src/task/processor.rs
pub fn run_tasks() {
    loop {
        check_futex_timer();
        let processor = get_proc_by_hartid(hart_id());
        // ... 从就绪队列获取任务并切换
    }
}
```

**缺失的多核调度机制**：
- **无负载均衡**：未找到任务迁移（task migration）、工作窃取（work stealing）逻辑
- **无 CPU 亲和性（affinity）**：未找到 `sched_setaffinity`、`cpu_mask` 等实现
- **无全局调度器锁**：由于单核设计，`PROCESSORS` 数组无需加锁访问

### 锁的实现与多核安全性

**结论：锁实现未考虑多核场景，无中断禁用 SpinLock，无优先级继承。**

1. **UPSafeCell（单处理器原语）**：
   ```rust
   // os/src/sync/up.rs
   pub struct UPSafeCell<T> {
       inner: RefCell<T>,
   }
   unsafe impl<T> Sync for UPSafeCell<T> {}
   ```
   - 基于 `RefCell` 的运行时借用检查
   - **仅适用于单核**，在多核下会产生数据竞争
   - 注释明确说明："We should only use it in uniprocessor"

2. **spin::Mutex（外部库）**：
   - 项目广泛使用 `spin::Mutex`（来自 `spin` crate）
   - **未实现中断禁用**：搜索 `disable_interrupt`、`cli`、`local_irq_save` 无结果
   - **无优先级继承**：未找到优先级继承（priority inheritance）或优先级天花板（priority ceiling）实现

3. **Futex 实现**：
   - `os/src/task/futex.rs` 实现了用户态 futex（快速用户态互斥锁）
   - 使用 `FUTEX_QUEUE_BITMAP`（`Mutex<BTreeMap>`）管理等待队列
   - 支持 `FUTEX_WAIT`、`FUTEX_WAKE`、`FUTEX_REQUEUE`、`FUTEX_WAIT_BITSET`
   - **多核行为**：futex 依赖原子操作和内核等待队列，在多核下可工作，但 `Mutex` 保护可能成为瓶颈

### 与前面章节的交叉引用

1. **进程调度中的全局唯一 ID 池**（第 4 章）：
   - `FUTEX_KEY_COUNTER` 使用 `AtomicUsize` 确保线程安全
   - PID/TID 分配使用 `id_allocator.rs`（未在此章详细分析）

2. **双级注册机制**（第 4 章）：
   - 线程注册到 `Process` + 全局 `TASK_MANAGER`
   - 在多核下需要锁保护，但当前使用 `spin::Mutex` 无中断禁用

3. **同步互斥中的 Futex 实现**（第 5 章）：
   - futex 等待队列由 `Mutex<BTreeMap>` 保护
   - 在多核场景下，多个 hart 可能同时访问队列，需要更细粒度锁

4. **原子操作**（第 5 章）：
   - 项目使用 `core::sync::atomic` 模块
   - 内存序使用 `Ordering::Relaxed`、`Ordering::SeqCst`，但未系统使用 `Acquire`/`Release` 语义

### 总结

| 特性 | 实现状态 | 说明 |
|------|----------|------|
| 多核支持 | ❌ 未实现 | `HART_NUM=1`，硬编码单核 |
| Secondary CPU 启动 | ❌ 未实现 | `boot_all_harts` 被注释 |
| IPI 机制 | ❌ 未实现 | 仅 lwext4 example 中有 SBI 封装 |
| Per-CPU 变量 | ⚠️ 单核可用 | `PROCESSORS` 数组，无锁访问 |
| 负载均衡 | ❌ 未实现 | 无任务迁移逻辑 |
| CPU 亲和性 | ❌ 未实现 | 无 `sched_setaffinity` |
| SpinLock 禁中断 | ❌ 未实现 | 使用 `spin::Mutex` 无中断控制 |
| 优先级继承 | ❌ 未实现 | 无相关代码 |
| UPSafeCell | ⚠️ 单核专用 | 基于 `RefCell`，多核不安全 |
| 原子操作 | ✅ 已实现 | `AtomicUsize` 用于计数器 |

**总体评价**：本项目为**单核操作系统**，所有同步机制和调度器设计均基于 UP 假设。若需扩展至多核，需：
1. 实现真正的 `hart_id()` 获取当前 CPU ID
2. 实现 Secondary CPU 启动流程（SBI `hart_start`）
3. 将 `UPSafeCell` 替换为多核安全的 `SpinLock`（禁用中断）
4. 实现 IPI 机制用于核间通信和 TLB shootdown
5. 添加多核调度策略（负载均衡、CPU 亲和性）
