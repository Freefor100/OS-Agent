### 调度算法与策略（代码证据）

本操作系统实现了**基于就绪队列的 FIFO（先进先出）调度策略**，**不支持优先级调度、时间片轮转或 CFS**。

#### 调度器实现

**文件路径**: `os/src/task/manager.rs:10-65`

```rust
pub mod ready_queue {
    static READY_QUEUE: Lazy<Mutex<VecDeque<Weak<TaskControlBlock>>>> =
        Lazy::new(|| Mutex::new(VecDeque::new()));

    pub fn add_task(task: &Arc<TaskControlBlock>) {
        let mut queue = READY_QUEUE.lock();
        if task_in_queue(&queue, task) {
            panic!("add task fail: task already in queue!");
        } else {
            queue.push_back(Arc::downgrade(&task));
        }
    }

    pub fn fetch_task() -> Option<Arc<TaskControlBlock>> {
        loop {
            if let Some(task) = READY_QUEUE.lock().pop_front() {
                if let Some(task_arc) = task.upgrade() {
                    return Some(task_arc);
                }
            } else {
                return None;
            }
        }
    }
}
```

**关键证据**：
1. **数据结构**：使用 `VecDeque`（双端队列）实现就绪队列，`add_task` 从尾部插入，`fetch_task` 从头部取出，符合 FIFO 语义。
2. **无优先级字段**：`TaskControlBlock` 和 `TaskControlBlockInner` 中**不存在** `priority`、`stride`、`sched_class` 等字段。
3. **调度策略系统调用**：`sys_sched_setscheduler` 和 `sys_sched_getscheduler` 均为桩函数（`os/src/syscall/process/others.rs:151-165`），注释明确说明"使用的是标准的时间片调度算法，直接返回 SCHED_OTHER = 0"。

**文件路径**: `os/src/syscall/process/others.rs:161-165`

**结论**：调度器为**简单 FIFO**，**未实现**优先级调度、RR（时间片轮转）、Stride 或 CFS（完全公平调度）。

### 上下文切换实现（汇编分析）

上下文切换通过汇编函数 `__switch` 实现，保存 callee-saved 寄存器到 `TaskContext` 结构体。

#### LoongArch64 实现

**文件路径**: `os/src/arch/loongarch64/qemu/asms/switch.S:9-36`

```asm
.globl __switch
__switch:
    # save kernel stack of current task
    st.d $sp, $a0, 8
    # save ra & s0~s8 of current execution
    st.d $ra, $a0, 0
    .set n, 0
    .rept 9
        SAVE_SN %n          # 保存 s0-s8 到 [a0 + (n+2)*8]
        .set n, n + 1
    .endr
    st.d $fp, $a0, (9+2)*8
    # restore ra & s0~s8 of next execution
    ld.d $ra, $a1, 0
    .set n, 0
    .rept 9
        LOAD_SN %n          # 从 [a1 + (n+2)*8] 恢复 s0-s8
        .set n, n + 1
    .endr
    ld.d $fp, $a1, 0x58
    # restore kernel stack of next task
    ld.d $sp, $a1, 8
    li.d $a0, 0             # 返回 0 表示正常切换
    jr $ra
```

**保存的寄存器**（共 11 个）：
- `ra`（返回地址）
- `sp`（内核栈指针）
- `s0-s8`（9 个 callee-saved 寄存器）
- `fp`（帧指针）

#### RISC-V 64 实现

**文件路径**: `os/src/arch/riscv64/qemu/asms/switch.S:9-30`

```asm
.globl __switch
__switch:
    sd sp, 8(a0)            # 保存 sp
    sd ra, 0(a0)            # 保存 ra
    .set n, 0
    .rept 12
        SAVE_SN %n          # 保存 s0-s11
        .set n, n + 1
    .endr
    ld ra, 0(a1)            # 恢复 ra
    .set n, 0
    .rept 12
        LOAD_SN %n          # 恢复 s0-s11
        .set n, n + 1
    .endr
    ld sp, 8(a1)            # 恢复 sp
    li a0, 0
    ret
```

**保存的寄存器**（共 14 个）：
- `ra`（返回地址）
- `sp`（内核栈指针）
- `s0-s11`（12 个 callee-saved 寄存器）

**关键特点**：
- **不保存 caller-saved 寄存器**（如 `a0-a7`、`t0-t6`），由编译器保证在调用 `switch()` 前保存。
- **不保存浮点/向量寄存器**，内核**不支持**浮点上下文切换。
- 返回值 `a0=0` 表示正常切换；若 `a0>0` 则表示被切换出的任务需要释放内核栈（用于 `__abandon`）。

```rust
    let mut task_inner = task.inner_lock();
    let signal = SigSet::from_sig(signo);
```

### 进程/线程管理模块扩展

#### POSIX 资源限制

**部分实现**：仅支持 `RLIMIT_NOFILE`（文件描述符数量限制）。

**文件路径**: `os/src/syscall/fs.rs:1072-1107`

```rust
pub fn sys_prlimit(pid: usize, resource: u32, new_limit: *const RLimit, old_limit: *mut RLimit) -> SyscallRet {
    const RLIMIT_NOFILE: u32 = 7;
    if resource != RLIMIT_NOFILE {
        return Ok(0);  // 其他资源类型直接返回成功
    }
    if pid == 0 {  // 仅支持当前进程
        let task = current_task().unwrap();
        let mut inner = task.inner_lock();
        let fd_table = &mut inner.fd_table;
        if !old_limit.is_null() {
            let lim = RLimit {
                rlim_cur: fd_table.get_soft_limit(),
                rlim_max: fd_table.get_hard_limit(),
            };
            memory_set.put_data(old_limit, lim);
        }
        if !new_limit.is_null() {
            let limit: RLimit = memory_set.get_data(new_limit);
            fd_table.set_limit(limit.rlim_cur, limit.rlim_max);
        }
    } else {
        unimplemented!("pid must equal zero");
    }
    Ok(0)
}
```

**限制**：
- **仅支持 1 种资源类型**（RLIMIT_NOFILE），POSIX 定义了 16 种（CPU 时间、文件大小、数据段等）。
- **不支持** `getrlimit()` / `setrlimit()` 独立系统调用，仅通过 `prlimit64` 实现。
- **仅支持**操作当前进程（pid=0），无法查询/设置其他进程的资源限制。

#### 线程与进程的区别

本操作系统中，**线程与进程共享相同的 TCB 结构**，区别仅在于：

| 特性 | 进程 | 线程 |
|------|------|------|
| **地址空间** | 独立 `MemorySet` | 共享父进程 `MemorySet`（CLONE_VM） |
| **文件表** | 可独立（!CLONE_FILES） | 共享（CLONE_FILES） |
| **信号表** | 独立 `SigTable` | 共享（CLONE_SIGHAND） |
| **PID** | 独立 PID | 与进程组 PID 相同（CLONE_THREAD） |
| **TID** | 独立 TID（=PID） | 独立 TID（≠PID） |

**关键代码**（`os/src/syscall/process/clone.rs:263-310`）：
```rust
if flags.contains(CloneFlags::CLONE_THREAD) {
    timer = Arc::clone(&parent_inner.timer);
    sig_mask = SigSet::empty();
    process = self.process.clone();  // 共享 Process
} else {
    process = Process::new(memory_set.clone(), sig_table.clone(), pid, ppid);  // 新 Process
}
```

**结论**：代码中**未区分** PCB 和 TCB，而是通过 `Process` 包含多个 `TaskControlBlock` 实现"进程-线程"层次结构。
