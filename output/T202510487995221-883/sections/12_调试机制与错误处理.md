## 日志与打印系统

### 日志宏实现

TatlinOS 使用 Rust 标准的 `log` crate 作为日志框架，日志功能在 `os/src/logger.rs` 中实现。日志系统支持 5 个级别，通过 Cargo features 在编译时配置：

```rust
// os/src/logger.rs:5-17
const LOG_LEVEL: log::LevelFilter = if cfg!(feature = "error") {
    log::LevelFilter::Error
} else if cfg!(feature = "warn") {
    log::LevelFilter::Warn
} else if cfg!(feature = "info") {
    log::LevelFilter::Info
} else if cfg!(feature = "debug") {
    log::LevelFilter::Debug
} else if cfg!(feature = "trace") {
    log::LevelFilter::Trace
} else {
    log::LevelFilter::Info // 默认级别
};
```

**日志级别设计**：
- `error`：错误级别
- `warn`：警告级别  
- `info`：信息级别（默认）
- `debug`：调试级别
- `trace`：追踪级别

### 日志输出格式

日志输出包含丰富的上下文信息，在 `SimpleLogger::log()` 中实现：

```rust
// os/src/logger.rs:22-35
fn log(&self, record: &Record) {
    let (pid, tid) = if let Some(task) = current_task() {
        (task.pid() as isize, task.tid() as isize)
    } else {
        (-1, -1)
    };
    if self.enabled(record.metadata()) {
        println!(
            "\x1b[{}m[{}] [HART{}] [PID {}] [TID {}] {}\x1b[0m",
            level_color(record.level()),
            record.level(),
            hart_id(),
            pid,
            tid,
            record.args()
        );
    }
}
```

**输出格式**：`[级别] [HART 核 ID] [PID 进程 ID] [TID 线程 ID] 消息内容`

日志支持 ANSI 颜色输出，不同级别使用不同颜色：
- Error: 红色 (31)
- Warn: 亮黄色 (93)
- Info: 蓝色 (34)
- Debug: 绿色 (32)
- Trace: 亮黑色 (90)

### 打印宏

内核提供 `print!` 和 `println!` 宏，定义在 `os/src/console.rs`，通过 SBI 调用实现字符输出：

```rust
// os/src/console.rs:8-12
impl Write for Stdout {
    fn write_str(&mut self, s: &str) -> fmt::Result {
        for c in s.chars() {
            console_putchar(c as u8);
        }
        Ok(())
    }
}
```

底层通过 `os/src/sbi.rs` 的 `console_putchar()` 调用 SBI 接口输出字符。

## Panic 处理与栈回溯

### Panic Handler 实现

Panic 处理器定义在 `os/src/lang_items.rs`，当 Rust 代码触发 `panic!` 时会被调用：

```rust
// os/src/lang_items.rs:22-35
#[panic_handler]
fn panic(info: &PanicInfo) -> ! {
    println!("panic");
    if let Some(location) = info.location() {
        println!(
            "[kernel] Panicked at {}:{} {}",
            location.file(),
            location.line(),
            info.message().unwrap()
        );
    } else {
        println!("[kernel] Panicked: {}", info.message().unwrap());
    }
    shutdown(true)
}
```

**Panic 处理流程**：
1. 打印 "panic" 标识
2. 输出 panic 位置（文件、行号、消息）
3. 调用 `shutdown(true)` 停机

### 停机实现

停机函数因架构而异：

**RISC-V 架构** (`os/src/arch/riscv64/qemu/cpu.rs:18-25`)：
```rust
pub fn shutdown(failure: bool) -> ! {
    if !failure {
        system_reset(Shutdown, NoReason);
    } else {
        system_reset(Shutdown, SystemFailure);
    }
    unreachable!()
}
```

**LoongArch64 架构** (`os/src/arch/loongarch64/qemu/cpu.rs:25-30`)：
```rust
pub fn shutdown(_failure: bool) -> ! {
    if cfg!(not(feature = "board")) {
        unsafe {
            ((0x9000_0000_100e_001c) as *mut u8).write_volatile(0x34);
        }
    }
    loop {}
}
```

在 QEMU 环境下，LoongArch64 通过写入特定 MMIO 地址触发关机；在板级环境下进入死循环。

### 栈回溯 (Backtrace) 支持情况

**关键发现**：TatlinOS 的栈回溯功能**仅有框架，未实现实际功能**。

在 `os/src/utils/mod.rs` 中定义了 `backtrace()` 函数：

```rust
// os/src/utils/mod.rs:16-30
pub fn backtrace() {
    // 算了，我们不应该指望这玩意能用
    // unsafe {
    //     let mut fp: usize;
    //     asm!("mv {}, fp", out(reg) fp);
    //     ... // 全部代码被注释掉
    // }
}
```

函数体为空，实现代码被完全注释掉，并附有注释"算了，我们不应该指望这玩意能用"。

**调用点分析**：
- `os/src/trap/mod.rs:157`：被注释掉的调用
- `os/src/trap/mod.rs:258`：在 `trap_from_kernel()` (RISC-V) 中调用
- `os/src/trap/mod.rs:279`：在 `trap_from_kernel()` (LoongArch64) 中调用

**结论**：
- **不支持 DWARF 解析**
- **不支持基于 FramePointer 的完整调用栈回溯**
- 当前实现仅在 panic 时打印 panic 位置信息，**不会打印完整的函数调用栈**
- 搜索 `dwarf`、`unwind`、`stack_trace` 等关键词未找到相关实现

## 错误码与 Result 设计

### 错误码定义

系统错误码定义在 `os/src/utils/error.rs`，使用 `SysErrNo` 枚举表示，采用标准 Linux 错误码：

```rust
// os/src/utils/error.rs:7-50
#[derive(Debug, Clone, Copy, FromPrimitive, PartialEq, Eq)]
#[repr(i32)]
pub enum SysErrNo {
    EUNDEF = 0,       // Undefined
    EPERM = 1,        // Operation not permitted
    ENOENT = 2,       // No such file or directory
    ESRCH = 3,        // No such process
    EINTR = 4,        // Interrupted system call
    EIO = 5,          // I/O error
    // ... 共 133+ 个错误码
    EHWPOISON = 133,  // Memory page has hardware error
    Default = 255,    // 未知错误
}
```

**错误码覆盖范围**：
- 文件操作错误（ENOENT、EISDIR、ENOTDIR 等）
- 进程管理错误（ESRCH、ECHILD 等）
- 内存错误（ENOMEM、EFAULT 等）
- 网络错误（ENETDOWN、ECONNRESET 等）
- 权限错误（EPERM、EACCES 等）

### Result 类型定义

系统调用返回值使用 `SyscallRet` 类型别名：

```rust
// os/src/utils/error.rs:426-428
pub type SyscallRet = Result<usize, SysErrNo>;
pub type GeneralRet = Result<(), SysErrNo>;
```

**返回值约定**：
- 成功：`Ok(返回值)`，用户态获取正数返回值
- 失败：`Err(错误码)`，用户态获取负数错误码

### 系统调用错误处理

在 `os/src/trap/mod.rs` 的 `trap_handler()` 中，系统调用返回值被转换为负数错误码：

```rust
// os/src/trap/mod.rs:85-94
cx.set_a0(match result {
    Ok(res) => res,
    Err(errno) => -(errno as isize) as usize,
});
```

## 调试接口与交互式 Shell

### 交互式 Shell 支持

**未发现内核级交互式 Shell 或 Monitor 实现**。

通过以下搜索确认：
- `grep "monitor|shell|command"` 仅找到构建脚本和 busybox 相关引用
- `grep "ps_main|ls_main|help"` 未找到 shell 命令实现
- 用户空间程序 `user/src/bin/` 目录下无 shell 程序

**用户空间 busybox 支持**：
- 文件系统初始化代码 `os/src/fs/kernel_fs_ops/initfiles.rs:246` 创建指向 `/musl/busybox` 的符号链接
- busybox 作为用户空间程序运行，**非内核调试接口**

### 调试控制台

内核通过 SBI 控制台提供基本打印功能，但**无交互式调试控制台**。

### 内核调试选项

通过 Cargo features 提供编译时调试选项（`os/Cargo.toml`）：

```toml
[features]
error = []          # 启用 error 级别日志
warn = []           # 启用 warn 级别日志
info = []           # 启用 info 级别日志（默认）
debug = []          # 启用 debug 级别日志
trace = []          # 启用 trace 级别日志
```

### Perf/Ftrace 支持

**未发现 perf 或 ftrace 支持**：
- 搜索 `perf|ftrace|tracepoint` 仅找到日志级别相关的 `trace` 关键词
- 未找到 tracepoints 插入代码
- 未找到性能分析基础设施

## GDB Stub 支持情况

**严格验证结果**：**不支持 GDB Stub**。

通过以下搜索确认：
- `grep "gdbstub|handle_gdb|gdb_packet"`：**未找到任何匹配**
- 无 GDB 数据包解析循环
- 无 GDB 协议处理代码

**说明**：虽然 `os/Cargo.toml` 的 `[profile.release]` 中设置了 `debug = true` 以保留调试信息用于 GDB 调试，但这仅表示**支持外部 GDB 连接**（通过 QEMU 的 GDB 服务器），**内核本身未实现 GDB Stub**。

## 本章总结

| 功能 | 支持状态 | 说明 |
|------|----------|------|
| 日志系统 | ✅ 已实现 | 支持 5 级别，编译时配置，带 PID/TID 上下文 |
| Panic 处理 | ✅ 已实现 | 打印位置信息后停机 |
| 栈回溯 (Backtrace) | ❌ 未实现 | 函数存在但代码被注释 |
| DWARF 解析 | ❌ 未实现 | 未找到相关代码 |
| 交互式 Shell | ❌ 未实现 | 仅用户空间 busybox |
| GDB Stub | ❌ 未实现 | 无数据包解析代码 |
| Perf/Ftrace | ❌ 未实现 | 无 tracepoints 支持 |
| 错误码设计 | ✅ 已实现 | 完整 Linux 兼容错误码 |
| 断言检查 | ✅ 已实现 | 广泛使用 assert! |

**核心结论**：
1. TatlinOS 具备基础的日志和 panic 处理能力，但**缺乏高级调试功能**
2. **栈回溯功能未实现**，panic 时仅打印 panic 位置，不打印调用栈
3. **无内核级调试接口**（无 Monitor、无 GDB Stub、无 Shell）
4. 错误处理机制完善，采用标准 Linux 错误码体系
5. 调试主要依赖外部工具（QEMU GDB 服务器）和日志输出
