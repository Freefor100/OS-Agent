# Markdown 输出格式规范

本文档定义了OS-Agent-D生成的技术分析报告的统一格式规范。

## 1. 标题层级规范

### ✅ 正确示例

```markdown
# 项目概览与技术栈

## 设计概览

这是一个基于Rust的操作系统...

### 核心特性

系统采用模块化设计...

#### 内存管理模块

内存管理使用伙伴系统...
```

### ❌ 错误示例

```markdown
# 项目概览与技术栈

# 再来一个一级标题（禁止重复！）

### ## 错误的嵌套标题

##### 跳过了四级标题（不要跳级！）
```

### 规则

- **一章一个一级标题**：每个.md文件只应该有一个`#`标题（由系统自动添加）
- **从二级开始**：LLM输出的内容从`##`开始
- **层级递进**：`##` → `###` → `####`，不跳级
- **最多四级**：避免使用五级标题及以下

---

## 2. 正文输出规范

### ✅ 正确示例

```markdown
## 任务调度机制

本系统采用多级反馈队列调度算法，主要特点包括...
```

### ❌ 错误示例

```markdown
现在我已经收集了足够的信息来撰写报告。让我开始编写...

## 关键发现总结

通过分析代码，我发现...

现在让我撰写完整的分析报告。

## 任务调度机制

本系统采用...
```

### 规则

- **直接输出内容**：不要包含LLM的"思考过程"
- **禁止准备文字**："现在我开始..."、"让我总结..."等
- **专注分析**：直奔主题，用事实说话

---

## 3. 代码块规范

### ✅ 正确示例

````markdown
```rust
pub struct TaskInner {
    id: TaskId,
    state: AtomicU8,
    ctx: UnsafeCell<TaskContext>,
    // ... 其他字段
}
```
````

### ❌ 错误示例

````markdown
```
// 没有标注语言！
pub struct TaskInner {
    ...1000行代码...  // 太长了！
}
```
````

### 规则

- **语言标注**：```rust、```c、```toml等
- **长度控制**：单个代码块≤50行
- **省略标注**：过长时用`// ...`注释省略
- **完整性**：确保代码片段可以理解上下文

---

## 4. 列表规范

### ✅ 正确示例

```markdown
### 核心特性

1. 内存安全：利用Rust的所有权系统
2. 零成本抽象：高级抽象不带来运行时开销
3. 并发安全：编译时保证数据竞争安全

### 实现方式

- 基于伙伴系统的物理内存分配器
- slab分配器用于小对象分配
- LRU算法实现内存回收
```

### ❌ 错误示例

```markdown
1. 第一点
* 混用有序和无序标记！
2. 第二点
- 又换了标记！
```

### 规则

- **有序列表**：`1.`、`2.`、`3.`
- **无序列表**：统一使用`-`或`*`
- **嵌套缩进**：2或4个空格

---

##5. 路径与代码元素引用

### ✅ 正确示例

```markdown
任务结构定义在`arceos/modules/axtask/src/task.rs`中，核心数据结构是`TaskInner`。
调度器通过`spawn_task()`函数创建新任务，并使用`sched_fifo`调度算法。
```

### ❌ 错误示例

```markdown
任务结构定义在arceos/modules/axtask/src/task.rs中（没反引号！）
核心数据结构是TaskInner（没反引号！）
```

### 规则

- **文件路径**：使用反引号，给出从repo根目录的完整相对路径
- **代码元素**：结构体、函数、变量、配置项都用反引号
- **函数调用**：带括号如`spawn_task()`

---

## 6. 图表引用

### ✅ 正确示例

```markdown
下图展示了系统的提交历史分布：

![每月提交量](charts/commits_monthly.png)

从图中可以看出...
```

### ❌ 错误示例

```markdown
图表：charts/commits_monthly.png

见附图

[图1] commits_monthly.png
```

### 规则

- **标准格式**：`![描述](path/to/image.png)`
- **描述性文字**：说明图表含义
- **相对路径**：从章节文件的位置算起

---

## 7. 专业术语

### ✅ 正确示例

- 页表（Page Table）
- 互斥锁（Mutex）
- 进程控制块（Process Control Block，PCB）
- 中断描述符表（IDT）

### ❌ 错误示例

- 分页表（错误翻译）
- 互斥（不完整）
- 进程控制块(PCB)（缺少空格）

### 规则

- **术语准确**：使用标准OS术语
- **中英混排**：留一个空格如`进程 (Process)`
- **首次出现**：给出完整名称和缩写

---

## 8. 章节结构建议

一个完整的章节建议包含：

```markdown
## 核心概念

简要说明本章要分析的内容

## 数据结构

### 主要数据结构

列出关键结构体/类

### 辅助数据结构

...

## 实现机制

### 机制1

原理 + 代码实现

### 机制2

...

## 关键代码分析

选取1-3个关键函数深入分析

## 小结

简要总结本章内容（可选）
```

---

## 总结

遵守这些格式规范可以：

✅ 提高报告可读性
✅ 保持专业性
✅ 便于后续处理（如转PDF）
✅ 与人类撰写的文档对比时得分更高

**关键原则**：简洁、清晰、专业、统一
