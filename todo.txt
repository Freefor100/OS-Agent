**💡 核心技术原理说明：**
* **为什么 QEMU 和物理板卡上的代码要修改？**
  虚拟机硬件与物理板卡在硬件机制、外设协议、时钟计数和并发处理上有根本不同：
  1. *内存容量与布局*：开发板物理内存有限且常与外设共享（如共用 64M/32M RAM），必须限制内核堆大小以防内存爆掉，QEMU 则无此顾虑。
  2. *外设与块驱动*：QEMU 通常使用简单的标准虚拟化设备 `VirtIO`；物理板卡必须针对实际芯片集编写物理 SPI/SDIO (SD卡接口) 驱动、AHCI/SATA 硬盘控制器驱动或 PCIe 驱动。
  3. *时钟基准*：硬件定时器 tick 频率由板载晶振/锁相环分频决定（如 24MHz），与 QEMU 模拟时钟完全不同，如果不修改时钟换算常数，系统时间（如 sleep API）将全部错乱。
  4. *缓存一致性与延迟*：真实硬件读写存在物理延迟，且通过 DMA 读写时必须手动执行 CPU 缓存刷写与失效（Cache Invalidation）指令，否则 CPU 会读到 Cache 里的旧脏数据。
  5. *真实多核并发*：物理多核并发存在硬核竞争与 Cache 一致性机制，必须加入内存屏障（如 fence、dbar）及精确的锁，否则在 QEMU 下表现正常的系统在硬件上会立即死锁。
* **什么是 `initproc` 及其修改逻辑？**
  `initproc` (PID 1) 是系统开机后拉起的第一个用户态进程，在常规 OS 中它负责拉起用户 Shell 并循环 `wait` 收割退出的孤儿进程以防其成为僵尸进程。但在操作系统比赛评测中，为了无人值守地进行自动评估，`initproc` 被修改为**自动测试运行器 (Test Runner)**，它直接循环 `fork-exec-wait` 跑完硬编码的所有测例。选手常常在 `initproc` 中增加新测例、放宽超时时限，或者屏蔽注释掉那些会在物理板上卡死的测例，以实现“刷分保分”。

## 3. Agent C 识别 OS 图谱 / 同源变体聚类与深层代码查重

经过对 2025 届 `Undefined OS`、`AstrancE` 等队伍以及 2023 届 `StarryOS` 原创作品的深度调研，明确了以下查重与架构分析路线：

1. **OS 图谱四大流派分类与架构生成**：
   参赛作品主要分为四大流派，OS-Agent C 需针对不同流派采取不同的架构提取与比对策略：
   - **ArceOS 派系（Unikernel 转 Monolithic）**：基于 `StarryOS` 模板（repos/starry-202310003101649）。特征为 `api/` -> `core/` -> `arceos/` 的三明治结构。Agent 需解析 `Cargo.toml` 工作区与 `use` 依赖，生成跨库调用拓扑图，重点查重其 Syscall 包装层。
   - **rCore / zCore 派系（Rust 宏/微内核）**：基于官方 Tutorial 演进。特征为单体紧耦合或强异步架构。Agent 需利用 `tree-sitter` 提取核心模块（`task`, `mm`, `fs`）的控制流图（CFG），对比其与 rCore 标准调用树的拓扑同构率。
   - **xv6 派系（C 语言宏内核）**：基于 MIT xv6。特征为 C 语言结构、传统的锁与调度机制。Agent 需通过分析 `.c/.h` 文件包含关系及函数 AST Hash，剥离 xv6 原生代码，定位选手新增的特性逻辑。
   - **纯自研派系（独立硬核作品）**：如 Eonix 等不依赖教学框架的自研内核。特征为独特的构建系统、定制的调度器甚至专有 Bootloader。Agent 需提取其**核心创新模块架构**，重点分析底层汇编 `.S` 和链接脚本 `.ld` 的异构性以彰显其原创工作量。

2. **多维度的显微镜级工作量评估（针对同源项目）**：
   - **基线比对（Baseline Diff）**：将各流派的官方原型（如 `starry-2023`, `rCore-Tutorial`, `xv6-riscv`）设为零点基线，剔除无修改的原生代码。
   - **检测点一（接口层/高级特性）**：对于高阶 Syscall（如 mmap, futex, epoll），比对封装逻辑、错误处理机制、以及并发锁粒度，捕捉代码块克隆。
   - **检测点二（底层侵入检测）**：检查参赛队伍是否对引用的底层框架（如 `axfs`, `virtio-drivers`）做了实际的定制与重构，区分“纯挂载调包”与“深度魔改”。
   - **检测点三（核心数据结构差异）**：分析进程控制块（PCB）、页表管理等关键结构布局（如使用 `BTreeMap` 还是原生数组），通过数据结构签名（Data Structure Signature）锁定无法通过换皮掩盖的实现差异。


- 用 Agent D 已产出的 `compare_index.json` 多层指纹做相似度计算：
  design_layer + relation_layer(调用图同构) + code_structure_layer(ast_shape_hash) + lineage_layer
- OS-Agent C 的目标查重报告需包含：
  《模块宏观架构图》、《底层侵入式修改占比》、《高阶 Syscall 代码相似度矩阵》及《独立创新代码解析》。
