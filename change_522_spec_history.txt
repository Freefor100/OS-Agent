Change Spec — 2026-05-22（基于对话历史）
==========================================

背景
----
对 OS-Agent 项目进行两轮审查，发现题单（describe_stage_qa/*.json）对 LLM 产物
和规则校验代码之间存在多处逻辑冲突。本次修改修复全部已识别冲突，共涉及 6 个文件。

---

## 一、core/evidence_verifier.py

### 1.1 精简 stub 正则（冲突 8）
原 _STUB_HINT_RE 包含 `return 0;`、`return -1;`、`Ok(0)` 等过宽匹配。
正常实现函数末尾也可能有 `return 0`，导致合法实现被误判为 stub。
修后只保留明确的占位符关键词：todo!、unimplemented!、panic!(、ENOSYS、ENOTSUP、unsupported。

### 1.2 修复 coverage_sufficient 短路绕过题单阈值（冲突 2/6）
原 _negative_search_covers_policy() 只要 neg["coverage_sufficient"] is True 就直接
返回 True，完全绕过题单定义的 minimum_keyword_coverage / minimum_directory_coverage。
Task Agent（LLM）只需在输出里声明 coverage_sufficient=true，或不填该字段（原默认值
为 True），题单阈值就永远不会被校验。

修后：
- _parse_negative_search_metadata 中 coverage_sufficient 默认值改为 False
- _negative_search_covers_policy 只有当题单未定义任何阈值（required_keywords 和
  required_dirs 均为空）时，才信任自声明的 coverage_sufficient=True；否则必须通过
  _coverage_ratio 实际计算覆盖率

### 1.3 删除 accepted_source_types 参数（冲突 9 关闭）
原设计允许题单通过 evidence_policy.accepted_source_types 豁免文档证据的 -0.30 惩罚。
但文档描述的是设计意图，不能证明代码实际行为，豁免机制设计前提有误。
删除 accepted_source_types 参数，文档/README 证据的惩罚现在无条件生效。

---

## 二、core/feature_schema_bank/__init__.py

### 2.1 删除 read_code_segment 出 STRONG_EVIDENCE_TYPES（冲突 1）
read_code_segment 是工具名，不是证据类型名。混入 STRONG_EVIDENCE_TYPES 后，
Task Agent 只需设置 evidence_type="read_code_segment" 即可让证据被当作强类型，
绕过真正的实现证据要求，支撑 implemented 结论。

### 2.2 删除 accepted_source_types_for_question() 函数
配合 evidence_verifier.py 的删除，整体移除该机制。

---

## 三、core/describe_graph.py

### 3.1 Schema Guard stub 门控改为草稿语义判断（冲突 8 深层）
原 Schema Guard 对 stub 的判断完全依赖 evidence_verifier 的正则匹配。
Task Agent 做了语义判断认为是 stub，但证据摘录里没有命中正则关键词时，
Schema Guard 会将 stub 降级为 unknown，覆盖了 LLM 的语义判断。

修后：
- 新增模块级函数 _stub_supported_by_drafts(drafts)：遍历 Task Agent 草稿，
  若任意草稿以 confidence=medium/high 判断为 stub，返回 True
- _enforce_answer_schema 接收 drafts 参数，当 Schema Guard 要将 stub 降级时，
  先调用此函数；若草稿支撑，保留 stub 并写入 note，不降级
- _parse_one_answer_candidate、_ask_one_question_json_with_retry、
  _assemble_stage、_coerce_or_fallback_json 均透传 drafts 参数

### 3.2 contract_only Review 短路（冲突 10）
Review Agent 识别出 contract_only（格式/schema 问题）时，原来仍会生成
Task Agent 补证据任务，但 Task Agent 无法修正格式问题，任务无效。
修后：finding_type == "contract_only" 且 score_evidence >= 0.75 时，
直接跳过生成任务，emit review_fix_skipped 事件，coercion 在下次 assembly 自动修正。

### 3.3 evidence_id 丢弃时 emit 事件（冲突 5）
_resolve_answer_evidence_refs 在 dropped_ids 非空时已写入 answer["notes"]，
但没有调 self.events.emit()，运行时无感知，events.jsonl 和 dashboard 看不到。
修后补充 emit("evidence_id_dropped", ...) 调用。

### 3.4 assembler_coercion 事件输出 orig→new 详情（第二轮 Problem 10）
原事件只记录被 coerce 的 qid 列表，无法判断 LLM 输出了什么、被改成了什么。
修后记录 pre/post coerce 值的映射，格式为 "qid: 'orig' → 'new'; ..."。

### 3.5 批量 Writer 提示补充约束（第二轮 Problem 2）
_stage_json_writer_prompt() 原来缺少两行约束，而单题 Writer 已有：
- 若该 fact 定义了 allowed_values，fact_answers[*].value 必须使用其中一个枚举值
- tri_state_impl 判定必须遵守 tri_state_rule 和 anti_examples

## 四、core/describe_json_qa.py

### 4.1 fact_answers[*].value 按 allowed_values coerce（冲突 4）
原 coerce_answers_payload_by_stage_qa() 只对顶层 answer["value"] 做枚举归一化，
不处理 fact_answers[*].value。若某 fact 有 allowed_values，LLM 输出不合规的值
会静默保留，与顶层 coercion 路径不对称。
修后对每个 fact 查找对应 spec 的 allowed_values，若存在则对 fa["value"] 同样
调用 _coerce_single_choice_value()。

### 4.2 short_answer/fill_in 字段回填追踪（冲突 3）
ensure_structured_value_for_question() 会把 LLM 漏填的 fact_key 字段静默补成
"unknown"，LLM 的遗漏被消化掉，调试时无法得知哪些字段是 LLM 填的、哪些是补的。
修后在调用前后对比 value dict 的 keys，将新增的"补 unknown"字段记录到 notes，
格式为 "Coercion: 以下 structured_facts 字段由 LLM 漏填，已回填 unknown: [...]"。

---

## 五、core/describe_stage_review.py

### 5.1 finding_type 非法值归一化（第二轮 Problem 5）
Review Agent 可能输出不在枚举范围内的 finding_type 值。
_build_targeted_review_fix_task() 直接使用该值做分支判断，非法值会导致逻辑走错路径。
修后在 coerce_review_payload() 中检查 finding_type 是否属于合法枚举
{missing_evidence, weak_evidence, wrong_evidence, duplicate_evidence, contract_only}；
若不属于，归一化为 "weak_evidence" 并将原值保存为 _original_finding_type。

---

## 六、core/task_agents.py

### 6.1 evidence_candidates schema 增加 tool_name 字段（第二轮 Problem 4）
原 _records_from_react_output() 构造 EvidenceRecord 时 tool_name 硬编码为
"task_react_agent"，丢失了实际调用工具的信息。evidence_verifier 对 lsp_* 和
read_code_segment 工具有加分逻辑，但因 tool_name 被覆盖而永远不会触发。
修后在 evidence_candidates schema 中新增 tool_name 字段，要求 LLM 填写实际工具名；
_records_from_react_output 优先使用 LLM 提供的值，回退到 "task_react_agent"。

### 6.2 系统提示要求 confidence 字段必填（第二轮 Problem 3）
draft_answers 的 confidence 字段若省略，_stub_supported_by_drafts() 会因
confidence 不在 {medium, high} 中而返回 False，导致 stub 草稿被 Schema Guard 拒绝，
与 Task Agent 的语义判断矛盾。
修后在 _TASK_AGENT_SYSTEM 中新增硬性要求：每条 draft_answers 必须填写 confidence
（high/medium/low），省略时系统默认 low，stub 草稿将被 Schema Guard 拒绝。
